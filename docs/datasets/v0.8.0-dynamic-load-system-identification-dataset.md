# Dynamic Load System Identification Dataset for the Solana Containerised Testbed v0.8.0

## Dataset status

This dataset is associated with the `v0.8.0 dynamic load system identification` stage of the Solana Containerised Testbed.

Planned persistent identifiers:

- Software DOI: TBD
- Dataset DOI: TBD

## Purpose

This dataset provides controlled dynamic-load experimental data for system identification of the Solana Containerised Testbed.

The main identification channel is:

    u_cmd(t) -> u_ach(t)

where:

- `u_cmd(t)` is the commanded transaction load, represented by `target_lambda(t)` / `u_cmd`;
- `u_ach(t)` is the achieved throughput measured during the experiment.

The dataset supports initial throughput-channel modelling:

- steady-state gain estimation;
- step-response analysis;
- recovery analysis;
- multi-step profile tracking;
- low-frequency sine-approximation / frequency-response analysis.

The dataset is not yet sufficient for latency-constrained MPC because `lat_p99` is missing in the current `collect_csv` outputs.

## Safe identification range

The confirmed safe identification range used in this dataset is:

    lambda = 32..128

The previously observed higher-load boundary behaviour is outside the scope of this dataset. In particular, `lambda=512` is excluded.

## Experiment classes

### M0: steady-state baselines

Runs:

- `v0.8.0-M0-L32-R2`
- `v0.8.0-M0-L64`
- `v0.8.0-M0-L128`

Purpose:

- verify steady throughput tracking at fixed load levels;
- provide baseline observations for `lambda=32`, `lambda=64`, and `lambda=128`.

### M1: upward step-response experiments

Runs:

- `v0.8.0-M1-S32-64`
- `v0.8.0-M1-S64-128`
- `v0.8.0-M1-S32-128`

Purpose:

- observe response to increasing commanded load;
- estimate settling behaviour under upward transitions.

### M2: step-down and recovery experiments

Runs:

- `v0.8.0-M2-D128-64`
- `v0.8.0-M2-D64-32`
- `v0.8.0-M2-D128-32`

Purpose:

- observe response to decreasing commanded load;
- estimate recovery and settling under downward transitions.

### M3: multi-step load profiles

Runs:

- `v0.8.0-M3-MS32-64-128-64-32`
- `v0.8.0-M3-MS32-128-64-128-32`

Purpose:

- evaluate tracking under multi-step load profiles;
- validate dynamic behaviour across repeated upward and downward transitions.

### M4: sine-approximation load profile

Run:

- `v0.8.0-M4-SINE-L80-A48-T720`

Profile:

    80 -> 104 -> 122 -> 128 -> 122 -> 104 -> 80 -> 56 -> 38 -> 32 -> 38 -> 56 -> 80

Nominal parameters:

- `lambda0 = 80`
- `A = 48`
- `period = 720 s`
- safe range: `32..128`

Purpose:

- provide a low-frequency stepwise sine-approximation profile;
- estimate initial frequency-response indicators for the throughput channel.

## CSV fields

The main raw dataset file is `collect_csv.csv`.

Expected core fields:

| field | meaning |
|---|---|
| `t_sec` | elapsed experiment time in seconds |
| `u_cmd` | commanded load / target lambda |
| `u_ach` | achieved throughput |
| `sent_total` | cumulative sent transactions, if present |
| `err_per_sec` | observed errors per second |
| `inflight` | in-flight transaction count |
| `lat_p99` | p99 latency field, currently missing in collected outputs |

## Main findings

The v0.8.0 aggregate dynamic-identification summary reports:

- all M0-M4 experiment classes completed successfully;
- the confirmed safe identification range is `lambda=32..128`;
- observed `err_per_sec` remained zero across the included experiments;
- throughput tracking is close to unity gain;
- step-like transitions settle within approximately one 5-second sample interval;
- the M4 sine-approximation profile shows near-unity low-frequency gain and very high correlation between commanded and achieved throughput.

Current repository-side conclusion:

- `overall_status = PASS`
- `throughput_channel_ready_for_initial_modelling = true`
- `latency_channel_ready_for_modelling = false`

## Known limitations

The main limitation is latency telemetry.

The current `collect_csv.csv` outputs do not contain usable `lat_p99` values. Therefore:

- this dataset supports throughput-channel system identification;
- this dataset does not yet support latency-constrained MPC;
- latency-aware control modelling requires fixing or extending telemetry collection.

Additional limitations:

- settling-time estimates are limited by the 5-second sampling resolution;
- the M4 sine experiment is a stepwise sine approximation, not a continuously varied sinusoidal command;
- the dataset is intentionally limited to the safe range `lambda=32..128`;
- `lambda=512` is excluded due to previously identified storage-pressure risk.
