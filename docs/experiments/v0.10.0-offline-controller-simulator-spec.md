# v0.10.0 offline controller simulator specification

## Purpose

This document specifies an offline controller simulator for the Solana Containerised Testbed throughput-control research line.

The simulator is not a live Kubernetes controller. It is a preparation tool for testing PID and MPC control logic before any closed-loop experiment is attempted.

The simulator should use the v0.9.0 offline throughput model and the v0.10.0 controller-preparation constraints.

## Simulated control channel

The simulated control channel is:

    r[k] -> u_cmd[k] -> u_ach[k]

where:

- `r[k]` is the throughput reference;
- `u_cmd[k]` is the commanded load;
- `u_ach[k]` is the simulated achieved throughput.

## Plant model

The simulator should support two plant modes.

### Mode 1: settled unity-gain plant

    y[k] = u[k]

This mode represents the settled throughput behaviour found in v0.9.0.

### Mode 2: first-order transition plant

    y[k+1] = a*y[k] + b*u[k]

with:

    a = 0.143846
    b = 0.856154

This mode represents the first recorded post-transition behaviour identified in v0.9.0.

Important limitation:

The coefficients should not be interpreted as sub-5-second dynamics. They describe the first recorded post-transition sample in the current dataset.

## Reference profiles

The simulator should initially support safe reference profiles only.

Recommended profiles:

1. constant reference:

       r[k] = 64

2. step reference:

       64 -> 96

3. multi-step reference:

       64 -> 96 -> 128 -> 96 -> 64

4. cautious lower-range profile:

       32 -> 64 -> 32

All references must remain within:

    32 <= r[k] <= 128

## Command saturation

All candidate controller outputs must be saturated before being applied to the plant:

    u_cmd[k] = min(max(u_raw[k], 32), 128)

The simulator must record both:

- `u_raw[k]`
- `u_cmd[k]`

This is required to detect saturation and anti-windup behaviour.

## PID candidate

The simulator should support at least P-only and PI-only modes.

### P-only

    u_raw[k] = u_ff[k] + Kp * e[k]

where:

    e[k] = r[k] - y[k]

and the default feed-forward term is:

    u_ff[k] = r[k]

Given the near-unity-gain plant, the feed-forward term is expected to dominate.

### PI

    I[k] = I[k-1] + Ts * e[k]

    u_raw[k] = u_ff[k] + Kp * e[k] + Ki * I[k]

Anti-windup is required when saturation is active.

Minimum anti-windup rule:

    if u_raw[k] is outside the safe bounds:
        do not update I[k]

## MPC candidate

The simulator should support a basic finite-horizon MPC formulation using:

    y[k+1] = a*y[k] + b*u[k]

with:

    a = 0.143846
    b = 0.856154

Objective:

    minimise:
        sum q * (y_pred[i] - r_pred[i])^2
      + sum rho * (u[i] - u[i-1])^2

Constraints:

    32 <= u[i] <= 128

Initial recommendation:

- small horizon first;
- no latency constraints;
- no storage-pressure constraints;
- no command above 128.

## Metrics

The simulator should report:

- RMSE tracking error;
- MAE tracking error;
- max absolute tracking error;
- mean signed tracking error;
- saturation count;
- saturation fraction;
- command variation;
- final tracking error;
- settling indicator where applicable.

## Failure criteria

A simulated controller candidate should be rejected if:

- it commands outside the safe interval before saturation too frequently;
- it remains saturated for most of the profile;
- it produces persistent tracking offset;
- it creates unnecessary command oscillation;
- it relies on latency feedback, because latency telemetry is not yet available;
- it assumes valid operation above `lambda = 128`.

## Required outputs

A future simulator implementation should produce:

- `results/v0.10.0/controller-preparation/offline-simulator-spec-summary.json`
- `results/v0.10.0/controller-preparation/offline-simulator-spec-summary.md`

A future v0.11.0 implementation may produce:

- PID dry-run summaries;
- MPC dry-run summaries;
- plots or CSV traces for `r[k]`, `u_raw[k]`, `u_cmd[k]`, and `u_ach[k]`.

## Out of scope

This specification does not include:

- live closed-loop execution;
- Kubernetes Job generation for closed-loop control;
- latency-constrained MPC;
- reinforcement learning;
- MARL;
- commands above `lambda = 128`.

## Methodological conclusion

The offline simulator should be used as a gate before live controller experiments.

A controller should not be tested against the live Solana testbed until it has passed offline checks for:

- safe command bounds;
- stable tracking behaviour;
- saturation handling;
- anti-windup behaviour for PI/PID;
- no dependency on unavailable latency telemetry.
