# v0.10.0 throughput controller preparation summary

## Purpose

This document summarises the `v0.10.0 throughput controller preparation` stage.

The stage prepares the Solana Containerised Testbed for future throughput-control research. It does not implement or deploy a live closed-loop controller.

## Starting point

The stage starts from the merged `v0.9.0 offline throughput modelling` results.

The v0.9.0 modelling established a simple throughput model for the safe range:

    lambda = 32..128

The modelling is based on:

- Software v0.8.0 DOI: `10.5281/zenodo.20828274`
- Dataset v0.8.0 DOI: `10.5281/zenodo.20834551`

## Control problem definition

The initial control objective is throughput tracking:

    r[k] -> u_ach[k]

where:

- `r[k]` is the desired throughput reference;
- `u_ach[k]` is the achieved throughput;
- `u_cmd[k]` is the manipulated command sent to the load generator.

The controlled variable is:

    y[k] = u_ach[k]

The manipulated variable is:

    u[k] = u_cmd[k]

## Plant model

The settled throughput model is:

    u_ach[k] ~= u_cmd[k]

The boundary-transition model is:

    y[k] = 0.143846 * y[k-1] + 0.856154 * u[k]

The transition coefficients describe the first recorded post-transition sample in the current dataset. They must not be interpreted as finely resolved sub-5-second dynamics.

## Safety policy

The initial safe command interval is:

    32 <= u_cmd[k] <= 128

The controller must apply command saturation before sending any command:

    u_cmd[k] = min(max(u_raw[k], 32), 128)

The following remain excluded from this stage:

- `lambda = 256`, except as a future cautious boundary/reference point;
- `lambda = 512`, due to previously identified storage-pressure risk.

## PID preparation

The PID baseline is limited to conservative preparation:

- start with P-only or PI-only;
- include saturation from the beginning;
- require anti-windup before live integral control;
- avoid aggressive derivative action.

No live PID experiment is included in this stage.

## MPC preparation

The initial MPC formulation uses:

    y[k+1] = a*y[k] + b*u[k]

with:

    a = 0.143846
    b = 0.856154

Initial constraint:

    32 <= u[k+i] <= 128

Latency constraints remain disabled until usable latency telemetry is available.

## Offline simulator specification

The stage defines an offline simulator gate before live controller experiments.

The simulator should test:

- safe command bounds;
- tracking behaviour;
- saturation handling;
- anti-windup behaviour;
- PID dry-run behaviour;
- MPC dry-run behaviour;
- independence from unavailable latency telemetry.

## Main conclusion

The v0.10.0 stage converts the v0.9.0 throughput model into a controller-preparation specification.

The project is now ready for a later `v0.11.0 throughput controller prototype` stage, where an offline simulator or dry-run controller can be implemented before any live closed-loop Kubernetes experiment.

## Release decision

This stage is an internal controller-preparation stage. It does not introduce a new dataset and does not run new Kubernetes experiments.

Therefore, it can be merged into `main` without creating a separate Zenodo release.
