# v0.10.0 throughput controller preparation

## Purpose

This stage prepares the Solana Containerised Testbed for future throughput control experiments.

The purpose is not to deploy a closed-loop controller yet. The purpose is to define the control problem, plant model, safety bounds, saturation policy, PID baseline structure, and MPC formulation using the results of the v0.9.0 offline throughput modelling stage.

## Control objective

The initial control objective is throughput tracking:

    r[k] -> u_ach[k]

where:

- `r[k]` is the desired throughput reference;
- `u_ach[k]` is the achieved throughput measured from telemetry.

The manipulated variable is:

    u_cmd[k]

where `u_cmd[k]` is the commanded load sent to the load generator.

## Initial plant model

The v0.9.0 results support a simple throughput model in the confirmed safe range:

    lambda = 32..128

### Settled throughput model

    u_ach[k] ~= u_cmd[k]

This implies an initial steady-state plant gain:

    K ~= 1

### Boundary-transition correction

The v0.9.0 transition analysis gives:

    alpha = 0.856154

The corresponding transition model is:

    y[k] = y[k-1] + 0.856154 * (u[k] - y[k-1])

or:

    y[k] = 0.143846 * y[k-1] + 0.856154 * u[k]

This should be treated as a first recorded post-transition model, not as a sub-5-second transient model.

## Controlled variable

The controlled variable is:

    y[k] = u_ach[k]

Primary measurement source:

    collect_csv.csv -> u_ach

## Manipulated variable

The manipulated variable is:

    u[k] = u_cmd[k]

The controller computes a candidate command:

    u_raw[k]

The applied command is saturated to a safe interval:

    u_cmd[k] = sat(u_raw[k], u_min, u_max)

## Initial safe bounds

The initial safe command bounds are:

    u_min = 32
    u_max = 128

These bounds follow the v0.8.0/v0.9.0 evidence base.

The following values remain excluded from controller preparation:

- `lambda = 256`, except as a cautious boundary/reference point in a future stage;
- `lambda = 512`, excluded due to previously identified storage-pressure risk.

## Saturation policy

The controller must never send a command outside:

    32 <= u_cmd[k] <= 128

Saturation rule:

    if u_raw[k] < 32:
        u_cmd[k] = 32

    if u_raw[k] > 128:
        u_cmd[k] = 128

    otherwise:
        u_cmd[k] = u_raw[k]

## PID baseline preparation

The initial PID baseline should be designed for throughput tracking only.

Error definition:

    e[k] = r[k] - y[k]

Candidate incremental PID structure:

    u_raw[k] = u_cmd[k-1]
             + Kp * (e[k] - e[k-1])
             + Ki * Ts * e[k]
             + Kd / Ts * (e[k] - 2e[k-1] + e[k-2])

where:

- `Ts` is the controller sampling interval;
- `Kp`, `Ki`, and `Kd` are PID gains.

Initial conservative recommendation:

- start with P-only or PI-only;
- avoid aggressive derivative action;
- include saturation from the beginning;
- add anti-windup before integral action is used in live control.

## Anti-windup requirement

Integral action must not accumulate unbounded error when the command is saturated.

Minimum anti-windup policy:

- if `u_raw[k]` exceeds bounds and saturation is active, freeze the integral term;
- alternatively use back-calculation anti-windup in a later implementation.

This is a design requirement before live PID experiments.

## MPC preparation

The initial MPC formulation should use the simple first-order throughput model:

    y[k+1] = a * y[k] + b * u[k]

with:

    a = 0.143846
    b = 0.856154

These coefficients should be interpreted as a boundary-transition model derived from the first recorded post-transition samples. For settled operation, the v0.9.0 evidence supports the simpler approximation:

    y[k] ~= u[k]

Therefore, the initial MPC design should avoid over-interpreting the boundary-transition model as a finely resolved sub-sample transfer function.

Initial MPC objective:

    minimise sum over horizon:
        q * (y[k+i] - r[k+i])^2
      + rho * (u[k+i] - u[k+i-1])^2

Subject to:

    32 <= u[k+i] <= 128

Optional future constraint:

    |u[k+i] - u[k+i-1]| <= delta_u_max

Latency constraints must remain disabled until usable latency telemetry is available.

## Sampling note

The transition coefficient `alpha = 0.856154` is based on the first recorded post-transition sample in the v0.8.0 dataset.

The observed boundary interval is approximately 35 seconds.

Therefore, controller-preparation work must distinguish between:

- telemetry collection sampling;
- command segment boundaries;
- future controller sampling interval.

A future live controller should start with a conservative sampling interval and should not assume that sub-5-second dynamics have been identified.

## Out of scope

This stage does not include:

- live closed-loop Kubernetes experiments;
- latency-constrained MPC;
- reinforcement learning;
- MARL;
- automatic expansion beyond `lambda = 128`;
- experiments at `lambda = 512`.

## Success criteria

This stage is successful if it produces:

- a documented control problem definition;
- a documented safe command policy;
- a documented PID baseline structure;
- a documented MPC formulation;
- a machine-readable controller-preparation summary;
- clear limitations and safety constraints for future closed-loop experiments.
