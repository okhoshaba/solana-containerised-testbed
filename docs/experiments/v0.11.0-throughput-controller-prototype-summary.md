# v0.11.0 throughput controller prototype summary

## Purpose

This document summarises the `v0.11.0 throughput controller prototype` stage.

The stage implemented an offline throughput-controller prototype for the Solana Containerised Testbed research line.

The prototype is a dry-run simulator. It does not access Kubernetes, does not send load-generator commands, and does not implement live closed-loop control.

## Simulated control channel

The offline simulated channel is:

    r[k] -> controller -> u_raw[k] -> saturation -> u_cmd[k] -> plant -> y[k]

where:

- `r[k]` is the reference throughput;
- `u_raw[k]` is the raw controller command;
- `u_cmd[k]` is the saturated command;
- `y[k]` is the simulated achieved throughput.

## Plant model

The main comparison uses the first-order transition plant:

    y[k+1] = a*y[k] + b*u_cmd[k]

with:

    a = 0.143846
    b = 0.856154

This model represents first recorded post-transition behaviour from the v0.9.0 offline throughput modelling stage.

The coefficients must not be interpreted as finely resolved sub-5-second dynamics.

## Safety bounds

All simulated commands are saturated to:

    32 <= u_cmd[k] <= 128

The simulator records both:

- `u_raw[k]`
- `u_cmd[k]`

This makes saturation behaviour explicit and auditable.

## Implemented controller modes

The simulator implements three controller modes.

### Feed-forward baseline

    u_raw[k] = r[k]

This is the simplest baseline implied by the near-unity settled throughput model.

### P-only controller

    e_feedback[k] = r[k] - y[k-1]
    u_raw[k] = r[k] + Kp * e_feedback[k]

The tested parameter is:

    Kp = 0.10

### PI controller with freeze anti-windup

    e_feedback[k] = r[k] - y[k-1]
    I_candidate[k] = I[k-1] + Ts * e_feedback[k]

If the candidate command is outside the safe command interval, the integral state is frozen.

The tested parameters are:

    Kp = 0.10
    Ki = 0.002
    Ts = 1.0

## Automated comparison setup

The automated comparison uses:

    profile = multistep
    plant = transition
    hold = 12

Reference sequence:

    64 -> 96 -> 128 -> 96 -> 64

## Comparison result

The automated comparison produced:

    Best by MAE:  p_kp_0_10
    Best by RMSE: pi_kp_0_10_ki_0_002

Observed metrics:

| Case | Controller | RMSE | MAE | Saturation count | Anti-windup freeze count |
|---|---|---:|---:|---:|---:|
| `feedforward` | `feedforward` | 1.200998 | 0.358430 | 0 | 0 |
| `p_kp_0_10` | `p` | 0.731300 | 0.188537 | 12 | 0 |
| `pi_kp_0_10_ki_0_002` | `pi` | 0.726439 | 0.204742 | 12 | 12 |

## Interpretation

The P-only controller is the best simple controller by MAE.

The conservative PI controller slightly improves RMSE, but worsens MAE relative to P-only.

The PI anti-windup mechanism works as intended: the freeze counter activates during the saturated upper-bound segment, preventing further integral accumulation when the candidate raw command would exceed the safe range.

## Methodological conclusion

The v0.11.0 stage successfully implements an offline controller prototype and an automated comparison runner.

However, the result does not yet justify live closed-loop Kubernetes control.

The main reasons are:

- the comparison covers only one reference profile;
- the plant model is still based on first recorded post-transition samples rather than finely resolved dynamics;
- latency telemetry remains unavailable for latency-constrained control;
- storage-pressure constraints are not yet part of the controller;
- no live safety supervisor has been implemented.

## Config-driven simulation extension

After the initial controller prototype, the simulator was extended with JSON config-driven execution.

The config-driven mode allows a simulation to define:

- simulation name;
- plant mode;
- sampling interval;
- safe command bounds;
- piecewise-constant reference profile;
- controller type;
- controller gains.

The intended entry point is:

    scripts/v0.11.0/offline-throughput-controller-sim.py --config configs/v0.11.0/controller-sim/multistep-pi.json

The first config-driven PI run is intentionally equivalent to the previous CLI PI run with:

    profile = multistep
    plant = transition
    Kp = 0.10
    Ki = 0.002
    Ts = 1.0
    hold = 12

This confirms backward compatibility of the CLI path and establishes a portable simulation-input format.

## Broader offline profile coverage

The simulator was extended with a profile-coverage runner to test controller behaviour across multiple safe reference profiles.

Profiles covered:

- `constant64`
- `step64-96`
- `multistep`
- `lower-range`
- `sine-approx`

The profile-coverage result strengthens the previous interpretation:

- P-only remains the best controller by MAE on dynamic profiles;
- conservative PI often slightly improves RMSE, but worsens MAE;
- anti-windup behaviour is observable during saturated operation;
- broader offline coverage still does not constitute live closed-loop readiness.

## Optional plotting extension

The simulator was also extended with optional PNG plotting.

The plotting path is deliberately non-critical:

- JSON and CSV outputs remain the primary scientific artefacts;
- PNG output is produced only when requested;
- missing `matplotlib` must not fail the simulation;
- a skipped plot is acceptable on minimal Python installations.

This keeps the simulator portable while still allowing visual inspection on machines where plotting libraries are available.

## Recommended next stage

The recommended next stage is:

    v0.12.0 closed-loop readiness review

That stage should decide whether the offline prototype is safe enough to justify a limited live closed-loop experiment.

Before live control, the project should add:

- broader offline reference profile coverage;
- automated pass/fail safety checks;
- explicit live-control exclusion checks in scripts;
- controller configuration files;
- closed-loop readiness criteria;
- rollback and emergency stop procedure.
