# v0.11.0 throughput controller prototype

## Purpose

This stage implements an offline throughput-controller prototype for the Solana Containerised Testbed research line.

The prototype is a dry-run/offline simulator. It must not send commands to the live Kubernetes testbed.

The purpose is to test controller logic before any live closed-loop experiment is attempted.

## Scope

This stage implements an offline simulator for:

- reference profiles;
- plant model simulation;
- command saturation;
- P-only controller dry-run;
- PI controller dry-run;
- optional MPC dry-run preparation.

Live command execution is explicitly out of scope.

## Control channel

The offline simulated channel is:

    r[k] -> controller -> u_raw[k] -> saturation -> u_cmd[k] -> plant -> y[k]

where:

- `r[k]` is the desired throughput reference;
- `u_raw[k]` is the raw controller command;
- `u_cmd[k]` is the saturated command;
- `y[k]` is the simulated achieved throughput.

## Plant modes

The simulator should support at least two plant modes.

### Unity plant

    y[k] = u_cmd[k]

This represents settled throughput behaviour.

### First-order transition plant

    y[k+1] = a*y[k] + b*u_cmd[k]

with:

    a = 0.143846
    b = 0.856154

This represents the boundary-transition model derived in v0.9.0.

## Reference profiles

Initial built-in profiles:

1. constant:

       64

2. step:

       64 -> 96

3. multistep:

       64 -> 96 -> 128 -> 96 -> 64

4. lower-range:

       32 -> 64 -> 32

All references must remain within:

    32 <= r[k] <= 128

## Safety bounds

All commands must be saturated to:

    32 <= u_cmd[k] <= 128

The simulator must record:

- reference `r[k]`;
- simulated output `y[k]`;
- raw command `u_raw[k]`;
- saturated command `u_cmd[k]`;
- tracking error `e[k]`;
- saturation flag.

## Controller modes

### Feed-forward baseline

    u_raw[k] = r[k]

This is the baseline implied by the near-unity-gain throughput model.

### P-only controller

    e[k] = r[k] - y[k]

    u_raw[k] = r[k] + Kp * e[k]

### PI controller

    e[k] = r[k] - y[k]

    I[k] = I[k-1] + Ts * e[k]

    u_raw[k] = r[k] + Kp * e[k] + Ki * I[k]

Anti-windup is required for PI mode.

Minimum anti-windup rule:

    if saturation is active:
        freeze integral update

### MPC status

MPC is prepared conceptually in v0.10.0, but it is not required as a fully implemented controller in the first v0.11.0 prototype. The required v0.11.0 controller modes are feed-forward, P-only and PI.

## Success criteria

This stage is successful if:

- the offline simulator runs without live Kubernetes access;
- all commands are saturated inside `32..128`;
- feed-forward, P-only and PI modes can be evaluated;
- anti-windup behaviour is represented for PI mode;
- machine-readable and human-readable summaries are produced;
- no live closed-loop execution is attempted.

## Out of scope

This stage does not include:

- live Kubernetes controller;
- load-generator command execution;
- real-time feedback loop;
- latency-constrained MPC;
- reinforcement learning;
- MARL;
- commands above `lambda = 128`.
