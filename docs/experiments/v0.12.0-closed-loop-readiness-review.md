# v0.12.0 closed-loop readiness review

## Purpose

The purpose of `v0.12.0 closed-loop readiness review` is to decide whether the Solana Containerised Testbed research line is ready to proceed from offline controller design and simulation toward a bounded, supervised, limited live closed-loop experiment.

This stage is a formal readiness gate.

It does not start a live controller. It does not perform a closed-loop experiment. It does not modify a Kubernetes cluster.

The central question is:

> Can the project safely and methodologically proceed toward a limited live closed-loop experiment?

The answer is intentionally conservative. Full autonomous live control is not approved at this stage.

## Non-goals

This stage explicitly does not:

- run a live controller;
- apply controller decisions to a Kubernetes cluster;
- start or change transaction load;
- execute `kubectl apply`;
- perform a live closed-loop control experiment;
- validate production-like autonomous behaviour;
- tune PID or MPC parameters online;
- claim that the controller is safe for unrestricted live operation.

The stage is offline, documentary, and review-based.

## Starting point

The starting point is the sequence of prior research stages that prepared the project for controller work:

- `v0.8.0 dynamic load system identification`;
- `v0.9.0 offline throughput model`;
- `v0.10.0 controller-preparation constraints`;
- `v0.11.0 throughput controller prototype`;
- `v0.11.11 controller simulation safety checks`.

These stages provide the basis for asking whether a live experiment may be considered. They do not, by themselves, justify unrestricted autonomous control.

Readiness dimensions

The review uses the following readiness dimensions.

Artefact readiness

This dimension checks whether the repository contains the versioned artefacts required to support a formal review.

Modelling readiness

This dimension checks whether offline throughput modelling artefacts are present.

A live controller should not be introduced without an offline model baseline or equivalent prior system-identification evidence.

Controller simulation readiness

This dimension checks whether controller-prototype and simulator artefacts are present.

The project should not proceed directly from a theoretical controller concept to live actuation.

Safety readiness

This dimension checks whether offline controller safety checks are present.

Safety readiness does not mean that live operation is safe. It means only that the offline controller has passed a preliminary safety review sufficient to consider a bounded next step.

Observability readiness

This dimension checks whether prior observability and controlled-load work is represented.

Live closed-loop control requires reliable telemetry. If telemetry is missing, stale, inconsistent, or insufficiently captured, the controller must not increase load.

Operational readiness

This dimension checks whether the project has a live runbook, manual abort procedure, and rollback procedure.

This dimension is critical because the first live experiment must be supervised and interruptible.

Reproducibility readiness

This dimension checks whether the stage itself can be reproduced using committed documentation, committed result artefacts, and an offline checker.

Decision classes

The review supports four decision classes.

NO_GO

The project is not ready for shadow-mode validation or live closed-loop control.

This decision is appropriate when core offline artefacts are missing.

GO_FOR_SHADOW_MODE_ONLY

The project may proceed to a live telemetry shadow-mode controller, but not to live actuation.

In shadow mode, the controller may read telemetry and compute recommended actions, but it must not apply those actions.

CONDITIONAL_GO_FOR_LIMITED_LIVE

The project has enough offline preparation to consider a bounded, supervised, short-duration live closed-loop experiment, but only after explicit safety, observability, abort, and rollback preconditions are satisfied.

This decision does not approve full autonomous control.

GO_FOR_LIMITED_LIVE

The project is ready for a bounded, supervised, limited live closed-loop experiment.

This decision still does not approve unrestricted autonomous control.

Review procedure

The review is performed by:

python3 scripts/v0.12.0/check-closed-loop-readiness.py

The script:

runs offline;
checks expected prior artefacts;
does not call Kubernetes;
does not start a controller;
does not run a load generator;
writes readiness-review.json.

The generated JSON records:

the decision;
the rationale;
checked artefacts;
missing or weak items;
required preconditions;
safety constraints;
abort conditions;
observability requirements;
reproducibility notes.
Safety envelope for any future live experiment

Any future live experiment must be bounded by a strict safety envelope.

Minimum required constraints:

maximum transaction-generation rate;
minimum and maximum controller output;
controller output rate limit;
controller cooldown interval;
short maximum experiment duration;
conservative initial setpoint;
no uncontrolled setpoint changes;
no increase in load after telemetry loss;
fail-closed controller behaviour;
explicit manual supervision.

The controller output must be clamped.

The controller must not be able to command unbounded load.

The controller must fail closed, not fail open.

Required observability before live control

Before live actuation, the following must be observable:

validator health;
RPC latency;
RPC error rate;
transaction generation rate;
p95 and p99 latency, where available;
pod restart count;
CPU saturation;
memory saturation;
ledger growth rate;
controller input;
controller output;
controller setpoint;
controller error;
controller saturation state;
experiment start and end timestamps.

A future live controller must not increase load when telemetry is missing, stale, or internally inconsistent.

Required abort conditions

A limited live experiment must define automatic and manual abort conditions before the experiment begins.

Minimum abort conditions:

validator health becomes degraded or unavailable;
RPC error rate exceeds the configured threshold;
p95 or p99 transaction latency exceeds the configured threshold;
pod restart count increases unexpectedly;
CPU or memory saturation exceeds the configured threshold;
ledger growth rate exceeds the configured threshold;
telemetry is missing, stale, or internally inconsistent;
controller output attempts to exceed configured bounds;
operator manually aborts the run.

The manual abort command must be documented and tested before live control.

Operational constraints

A future live experiment must have:

a live experiment runbook;
a manual abort procedure;
a rollback procedure;
a maximum experiment duration;
a post-run audit procedure;
a clear definition of who supervises the experiment;
a clear definition of what data is collected;
a clear definition of what constitutes failure.

The first live experiment should be short, conservative, and supervised.

Reproducibility constraints

The review must be reproducible from committed repository artefacts.

The generated JSON should be committed together with this document and the result README.

The stage must remain offline-only. Its output is a review result, not a live experiment result.

Review result

The expected conservative result for the current stage is:

CONDITIONAL_GO_FOR_LIMITED_LIVE

This means:

the project is not approved for full autonomous live closed-loop control;
the project may consider a bounded, supervised, short-duration live experiment only after preconditions are satisfied;
shadow-mode validation remains the preferred next step.

The exact generated result is recorded in:

results/v0.12.0/closed-loop-readiness-review/readiness-review.json
Recommendation for next stage

The recommended next stage is:

v0.13.0 closed-loop shadow-mode controller validation

In this stage, the controller should read live telemetry and compute recommended actions, but not apply those actions.

A limited live closed-loop experiment should follow only after shadow-mode validation and after explicit safety, observability, abort, and rollback preconditions are satisfied.
