# v0.13.0 closed-loop shadow-mode controller validation

## Purpose

The purpose of v0.13.0 is to create and validate a config-driven shadow-mode controller runner.

This is not a live controller stage.

The runner computes controller recommendations but does not apply them.

## Non-goals

This stage does not:

- start a live controller
- call kubectl
- modify Kubernetes
- change transaction load
- perform live closed-loop control
- tune MPC
- approve autonomous control

## Method

The runner uses a config-driven offline replay workflow:

    configs/v0.13.0/shadow-controller-replay.json
    scripts/v0.13.0/run-shadow-controller.py

The runner searches for replay CSV files from earlier result directories. If a suitable replay CSV is found, it uses that file. If no suitable replay CSV is found, it falls back to a deterministic synthetic replay profile.

The plant prediction uses a discrete first-order model:

    y_hat[k+1] = a_y * y_hat[k] + b_u * u_cmd[k-delay]

The controller computes shadow recommendations using the configured controller mode.

By default, the controller mode is P.

PI and PID are supported structurally, but they should be introduced gradually only after replay validation.

## Offline-only guarantee

The runner is offline-only.

It does not call Kubernetes.

It does not start a controller.

It does not apply controller recommendations.

It does not generate transaction load.

The actuator_applied field is always false in the generated CSV and JSON.

## One-second simulation constraint

The default config uses sample_time_seconds equal to 1.0.

This is scientifically valid only if the discrete plant coefficients correspond to a one-second sample time.

If the sample time changes, plant coefficients must be recalculated.

## Outputs

The mandatory outputs are:

    results/v0.13.0/shadow-mode-validation/shadow-controller-timeseries.csv
    results/v0.13.0/shadow-mode-validation/shadow-controller-summary.json

CSV and JSON are mandatory.

Plots are intentionally not required in this stage.

## Scientific interpretation

The key scientific validation mode is replay validation.

The preferred input is real replay data from v0.8.0 or later containing a command signal such as u_cmd and an achieved signal such as u_ach.

When achieved output is available, the runner computes prediction error metrics.

When achieved output is not available, the run is still useful as a software and reproducibility check, but it is not sufficient for model-quality claims.

## Relationship to v0.12.0

v0.12.0 concluded CONDITIONAL_GO_FOR_LIMITED_LIVE.

That decision does not approve live control.

v0.13.0 is therefore intentionally conservative: it validates shadow-mode logic before any live actuation is considered.

## Result criterion

The stage is successful if:

- the runner executes offline
- it writes CSV output
- it writes JSON output
- actuator_applied is false
- no Kubernetes modification is performed
- the result can be reproduced from committed config and script files

## Recommended next stage

After v0.13.0, the recommended next step is to improve replay validation against real v0.8.0 command and achieved throughput data.

Only after replay validation should PI and PID behaviour be expanded.

MPC remains out of scope until the config-driven replay simulator is stable.
