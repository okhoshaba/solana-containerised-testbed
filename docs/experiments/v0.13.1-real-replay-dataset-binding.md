# v0.13.1 real replay dataset binding and validation

## Purpose

The purpose of v0.13.1 is to determine whether the repository already contains real replay datasets suitable for offline shadow-mode controller validation.

This stage does not extend the controller and does not move the research line toward live telemetry. It is a dataset-binding validation step between the config-driven v0.13.0 shadow-mode runner and any later real replay validation stage.

## Why v0.13.0 is not yet real replay validation

v0.13.0 introduced a config-driven offline shadow-mode controller runner.

However, the successful v0.13.0 run used this replay source:

    results/v0.11.0/controller-prototype/offline-controller-traces.csv

That source is useful for validating runner mechanics, but it is not sufficient evidence that the runner has been bound to a real v0.8.0 dynamic-load replay dataset.

## What is checked

The inspection script scans:

- results/v0.8.0/**/*.csv as preferred real replay candidates;
- results/v0.6.0/**/*.csv and results/v0.7.0/**/*.csv as secondary candidates.

For each CSV file, it records:

- CSV header;
- row count;
- matched command column, if present;
- matched achieved-throughput column, if present;
- matched time column, if present;
- whether the file is suitable for offline shadow replay.

A file is considered suitable when it is readable, non-empty, and contains both a command column and an achieved-throughput column.

## What is not run

This stage is offline-only.

It does not:

- call kubectl;
- change Kubernetes resources;
- start a live controller;
- generate transaction load;
- apply controller recommendations;
- execute closed-loop control;
- use live telemetry.

## Decision: REAL_REPLAY_BINDING_CANDIDATE_FOUND

REAL_REPLAY_BINDING_CANDIDATE_FOUND means that at least one suitable CSV file was found under results/v0.8.0.

In this case, the script writes a recommended_binding selected only from v0.8.0 candidates.

This decision does not mean that closed-loop control is safe. It only means that a candidate real replay source exists for a later offline shadow-mode replay run.

## Decision: REAL_REPLAY_BINDING_NOT_AVAILABLE

REAL_REPLAY_BINDING_NOT_AVAILABLE means that no suitable v0.8.0 CSV file was found with the minimum required command and achieved-throughput columns.

This is not a controller failure. It means the project should not claim real v0.8.0 replay validation yet.

## Why absence of a suitable v0.8.0 replay source is not a failure

The v0.13.1 gate is intentionally conservative. Its role is to prevent accidental over-claiming.

If no suitable v0.8.0 replay source exists, the correct research result is to record that real replay binding is not yet available.

## Reproduction

Run:

    python3 scripts/v0.13.1/inspect-replay-datasets.py --config configs/v0.13.1/real-replay-binding.json

Expected outputs:

- results/v0.13.1/real-replay-dataset-binding/replay-dataset-inventory.json
- results/v0.13.1/real-replay-dataset-binding/replay-binding-summary.json
