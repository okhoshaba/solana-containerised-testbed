# v0.13.5 shadow-mode refit comparison

## Purpose

v0.13.5 repeats shadow-mode validation with fitted/refit coefficients and compares the v0.13.0 placeholder model against the v0.13.4 fitted model.

The goal is to determine whether the fitted coefficients improve shadow-mode predictive quality and whether they materially change controller recommendations or introduce safety regressions.

## Scope

This stage is offline-only.

It does not:

- call kubectl;
- modify Kubernetes;
- start a live controller;
- generate transaction load;
- apply controller recommendations;
- execute closed-loop control;
- use live telemetry.

## Inputs

The replay input is:

    results/v0.13.2/real-replay-shadow-mode-validation/real-replay-shadow-timeseries.csv

The placeholder model source is:

    configs/v0.13.0/shadow-controller-replay.json

The fitted model source is:

    results/v0.13.4/refit-throughput-model/refit-coefficients.json

The v0.13.4 fitted model was derived from the same real replay line using one-step least-squares fitting.

## Method

The comparison uses an A/B replay protocol.

Both models are evaluated against the same replay input, controller policy, safety policy, and output schema.

The only intended difference between the two runs is the plant model:

1. v0.13.0 placeholder model;
2. v0.13.4 fitted/refit model.

For each replay row, the script records:

- observed replay output;
- replay command input;
- placeholder prediction;
- fitted prediction;
- prediction delta;
- placeholder recommended action;
- fitted recommended action;
- action delta;
- action disagreement flag;
- safety status for both models;
- fitted-new-safety-regression flag;
- actuator-applied flag.

The actuator-applied flag is always false.

## Decision

The stage decision is:

    caution

Reason:

The fitted model does not improve recursive/free-run shadow-mode predictive performance. It has substantially worse RMSE and MAE than the placeholder model, and it changes every controller recommendation in the evaluated replay.

The fitted model does not introduce safety regressions under the configured safety policy, but the prediction and action-difference results are too weak for promotion.

## Interpretation

v0.13.4 produced a strong one-step-ahead fit, but v0.13.5 shows that the same coefficients do not behave well when used recursively as a shadow-mode plant model.

This distinction is important:

- one-step fit evaluates local prediction using observed previous output;
- recursive/free-run simulation evaluates whether the model remains useful when its own previous predictions are fed forward.

The current fitted no-intercept model should therefore not be promoted as the default model candidate for closed-loop preparation.

## Recommended next step

The recommended next stage is a revised model-identification pass before further controller-preparation claims.

Candidate directions:

1. fit an intercept model:

       y[k+1] = c + a_y*y[k] + b_u*u[k-delay]

2. fit an equilibrium-deviation model:

       y[k+1] - y_eq = a_y*(y[k] - y_eq) + b_u*(u[k-delay] - u_eq)

3. evaluate both one-step and recursive/free-run validation metrics;

4. use a train/validation split or holdout replay segment before accepting fitted coefficients.

Closed-loop preparation should remain blocked until a fitted model improves recursive replay performance without safety regressions.
