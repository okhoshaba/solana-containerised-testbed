# v0.13.6 recursive plant model refit

## Purpose

v0.13.6 performs a recursive plant model refit after v0.13.5 showed that the v0.13.4 one-step fitted model did not behave well in recursive/free-run shadow-mode validation.

The central goal is to select a plant model using validation recursive RMSE rather than only one-step-ahead prediction quality.

## Background

v0.13.4 produced a fitted no-intercept model from real replay data with strong one-step metrics.

v0.13.5 then compared the v0.13.0 placeholder model against the v0.13.4 fitted model under recursive shadow-mode replay. The v0.13.4 fitted model performed worse than the placeholder model in recursive/free-run mode.

This established a methodological distinction:

- one-step fit evaluates local prediction using observed previous output;
- recursive/free-run validation evaluates whether model predictions remain useful when the model feeds its own previous predictions forward.

v0.13.6 is designed to close that gap.

## Scope

This stage is offline-only.

It does not:

- call kubectl;
- modify Kubernetes;
- start a live controller;
- generate transaction load;
- apply controller recommendations;
- execute closed-loop control;
- use live telemetry.

## Method

The script evaluates three model families.

### No-intercept model

    y[k+1] = a_y*y[k] + b_u*u[k-delay]

### Intercept model

    y[k+1] = c + a_y*y[k] + b_u*u[k-delay]

### Equilibrium-deviation model

    y[k+1] - y_eq = a_y*(y[k] - y_eq) + b_u*(u[k-delay] - u_eq)

Candidate delays:

    0, 1, 2, 3, 4, 5, 10, 15, 20

The replay contains 78 valid rows.

The train/validation split is:

- train rows: 50
- validation rows: 28
- train fraction: 0.65

The selection metric is:

    validation_recursive_rmse

## Selected model

The selected model is:

- family: `no_intercept`
- `c = 0.0`
- `a_y = 0.10889615191767432`
- `b_u = 0.8845312889194352`
- `delay_steps = 0`
- `y_eq = 97.16156286`
- `u_eq = 97.04`

The selected model equation is therefore:

    y[k+1] = 0.10889615191767432*y[k] + 0.8845312889194352*u[k]

## Decision

The stage decision is:

    pass

Reason:

The selected model improves recursive validation metrics versus both the v0.13.0 placeholder model and the v0.13.4 refit model under the configured thresholds.

## Interpretation

v0.13.6 confirms that the poor v0.13.5 result was not merely a failure of the replay pipeline. It was a model-selection problem: the v0.13.4 coefficients were selected by one-step fit quality, but they were not suitable for recursive/free-run simulation.

The v0.13.6 selected model performs much better under recursive validation.

However, the selected model is a zero-delay no-intercept model with a large command coefficient. This suggests that it may be capturing a strong apparent command-following relationship in the replay data. It should therefore be treated as a candidate shadow-mode plant model, not as proof of safe closed-loop controllability.

## Scientific conclusion

The key conclusion is:

    recursive validation must be a required model-selection gate before controller-preparation claims.

v0.13.6 provides a better candidate plant model for offline shadow-mode comparison, but it does not approve live control.

## Recommended next step

The next stage should be:

    v0.13.7 repeat shadow-mode comparison with selected recursive model

That stage should compare:

1. v0.13.0 placeholder model;
2. v0.13.4 one-step fitted model;
3. v0.13.6 selected recursive model.

The goal of v0.13.7 should be to determine whether the selected recursive model also improves shadow-mode controller recommendation quality without introducing safety regressions.
