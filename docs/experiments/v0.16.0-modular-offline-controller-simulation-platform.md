# v0.16.0 modular offline controller simulation platform

## Purpose

v0.16.0 builds a modular offline simulation platform for controller evaluation in the Solana Containerised Testbed throughput-control research line.

The main goal is not to improve MPC immediately. The main goal is to create a reusable simulation platform where the controller is a replaceable module.

The v0.14.3 P-only controller is used as the current best offline reference controller.

## Motivation

Previous MPC-oriented stages showed that the surrogate/weighted MPC line became comparable to the v0.14.3 P-only baseline, but did not demonstrate stable superiority on the current calibrated offline model.

Therefore, the next useful research step is to stop controller-specific tuning and build a common simulation platform.

This platform should later allow fair comparison of:

- v0.14.3 P-only reference controller;
- PID variants;
- surrogate MPC policies;
- strict MPC optimizers;
- adaptive controllers;
- heuristic policies.

## Reference controller

The v0.14.3 P-only controller is fixed as the current best offline reference controller on the current calibrated offline model.

For v0.16.0, this controller is not treated as a final production controller. It is used as a stable reference point for validating the simulation platform.

## Architecture

The v0.16.0 platform should separate the following responsibilities:

1. Simulation engine
2. Plant/model adapter
3. Controller interface
4. Scenario runner
5. Metrics evaluator
6. Artefact writer
7. Integrity checks

The simulation engine must not contain controller-specific logic.

A controller should be replaceable without rewriting the simulation loop.

## Controller interface contract

Each controller must expose a common interface:

- reset initial controller state before a scenario;
- receive an observation at each simulation step;
- return an action for the next plant/model step.

The observation should contain at least:

- current time step;
- current throughput;
- target throughput;
- current replicas;
- current control error;
- scenario id;
- profile id.

The action should contain at least:

- desired replicas;
- replica delta;
- raw control signal;
- clipped control signal;
- controller metadata.

## Initial supported controller

v0.16.0 initially supports only one controller:

- reference_p_only_v0.14.3

The purpose is to validate the platform architecture first, not to benchmark many controllers immediately.

## Inputs

The initial implementation should be driven by:

- configs/v0.16.0/modular-offline-simulation-platform.json

The configuration should define:

- stage name;
- reference controller;
- simulation horizon;
- replica constraints;
- enabled controllers;
- output directory;
- expected output artefacts.

## Outputs

The expected output directory is:

- results/v0.16.0/modular-offline-simulation-platform/

Expected artefacts:

- README.md
- simulation-summary.json
- controller-comparison.csv
- case-metrics.csv
- profile-metrics.csv
- reference-controller-metrics.json
- timeseries.csv

## Metrics

The first platform version should compute simple and reproducible controller metrics:

- mean absolute error;
- max absolute error;
- total absolute replica changes;
- number of replica changes;
- min replicas;
- max replicas;
- mean replicas;
- constraint violations.

The goal is not to create final scientific metrics in v0.16.0. The goal is to create a stable metrics layer that can be extended later.

## Safety constraints

The simulator must apply controller-independent safety constraints:

- minimum replicas;
- maximum replicas;
- maximum replica change per step;
- valid numeric outputs;
- no negative replica count;
- no missing output metrics.

Safety constraints should be enforced outside the controller when possible.

## Non-goals

v0.16.0 does not attempt to:

- improve MPC;
- tune MPC weights;
- implement strict MPC optimization;
- introduce adaptive control;
- run live Kubernetes experiments;
- change the calibrated offline model;
- perform large controller benchmarking.

## Acceptance criteria

v0.16.0 is considered successful if:

1. The platform has a separate simulation engine.
2. The controller is implemented as a replaceable module.
3. v0.14.3 P-only baseline is integrated as the reference controller.
4. The simulation run is reproducible from config.
5. The platform writes JSON and CSV artefacts.
6. Per-case and per-profile metrics are generated.
7. Integrity checks pass.
8. Documentation explains architecture, inputs, outputs, limitations, and next steps.

## Expected follow-up stages

After v0.16.0, possible follow-up stages are:

- v0.16.1 strict controller interface validation;
- v0.16.2 additional PID variants;
- v0.17.0 strict MPC optimizer integration;
- v0.18.0 adaptive controller experiments.
