# v0.6.0 Controlled Load Experiments

## Purpose

This stage introduces repeated controlled transaction-load experiments into the Solana Containerised Testbed.

The goal of this stage is not yet to determine the maximum throughput of the system. The goal is to establish a repeatable experimental workflow, collect structured telemetry, and verify that multiple runs under comparable load conditions produce analysable data.

## Current methodological position

The clean-server deployment automation validation stage, previously planned as v0.5.1, is temporarily deferred because of technical problems on the dedicated clean validation server.

Therefore, v0.6.0 is executed on the existing Dell server using the already validated Kubernetes-based Solana testbed.

The results of this stage validate controlled-load experimentation on the Dell-based environment. They do not yet claim clean-server reproducibility.

## Main research question

How does the containerised Solana testbed behave under repeated controlled transaction load when observed through Kubernetes, RPC, transaction telemetry, and host-level resource metrics?

## Experimental focus

This stage focuses on:

- repeated experiment execution;
- run metadata capture;
- host and Kubernetes preflight capture;
- controlled transaction-load execution;
- transaction success and failure accounting;
- latency and throughput observation;
- preparation of data for later saturation and knee-point analysis in v0.7.0.

## Expected run structure

Each experimental run should contain:

1. preflight capture;
2. warm-up period;
3. controlled transaction-load period;
4. telemetry capture;
5. post-run validation;
6. cool-down period;
7. structured result storage.

## Minimum metadata per run

Each run should record:

- run_id;
- date and time in UTC;
- hostname;
- Git branch;
- Git commit hash;
- Kubernetes node state;
- pod state;
- service state;
- Solana RPC endpoint;
- load level;
- target transaction rate;
- submitted transactions;
- successful transactions;
- failed transactions;
- duration;
- notes.

## Initial load levels

The initial load levels for v0.6.0 are:

| Level | Meaning |
|---|---|
| L0 | Idle baseline |
| L1 | Low controlled load |
| L2 | Moderate controlled load |
| L3 | Elevated but non-saturating controlled load |

The exact numerical transaction rates will be selected after the Dell preflight and pilot runs.

## Success criteria

The stage is considered successful if:

1. The repository contains a dedicated v0.6.0 branch.
2. The experiment directory structure is present.
3. The Dell server preflight state is captured.
4. The Kubernetes state is captured before load testing.
5. The Solana RPC endpoint is checked before load testing.
6. Repeated controlled-load runs can be executed.
7. Each run produces structured raw data and metadata.
8. Summary files can be generated from the raw results.
9. The results can later support v0.7.0 saturation and knee-point analysis.

## Limitation

Because v0.5.1 clean-server deployment validation is deferred, this stage validates controlled-load experimentation on the existing Dell server only. Full clean-server reproducibility remains a separate later validation stage.
