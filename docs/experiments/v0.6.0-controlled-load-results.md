# v0.6.0 Controlled Load Experiments: Results

## Purpose

This document summarises the results of the v0.6.0 repeated controlled-load experiment stage of the Solana Containerised Testbed.

The purpose of this stage was not to identify the saturation point of the system. The purpose was to validate that controlled transaction-load experiments can be executed repeatedly, captured systematically, and summarised at both run and level granularity.

Saturation and knee-point analysis are reserved for the later v0.7.0 stage.

## Experimental context

The experiments were executed on the Dell-based Kubernetes environment used by the Solana Containerised Testbed.

The tested stack included:

- Kubernetes-based Solana testbed;
- solana-validator workload;
- solana-monitor workload;
- solana-loadgen2 controlled load generator;
- Solana RPC service exposed inside Kubernetes as solana-rpc;
- RPC access from the Dell host through kubectl port-forward;
- load generator control through the solana-loadgen2 service.

The observed Solana version during the preflight checks was:

    solana-core: 1.18.25

## Load generator interface

The loadgen2 component exposes a simple HTTP control interface:

    GET  /health
    GET  /stats
    POST /rate

The load level is controlled by sending a JSON payload to /rate:

    {"lambda": 1}

At the end of every controlled-load run, the run script returns the generator to:

    target_lambda=0

This is an important safety condition for repeatable experiments.

## Experiment matrix

The completed v0.6.0 matrix is:

| Level | Target lambda | Meaning | Repetitions |
|---|---:|---|---:|
| L0 | 0 | idle baseline | 1 |
| L1 | 1 | low controlled load | 5 |
| L2 | 2 | moderate controlled load | 5 |
| L3 | 4 | elevated controlled load | 5 |

The repeated-load levels therefore contain 15 controlled-load runs in total.

## Per-level summary

The processed per-level summary is stored in:

    results/v0.6.0/processed/v0.6.0-level-summary.csv

The observed summary was:

| Level | Target lambda | Run count | Clean runs | Mean accepted delta | Error delta total | Rate estimate |
|---|---:|---:|---:|---:|---:|---:|
| L0 | - | 1 | 1 | - | - | - |
| L1 | 1 | 5 | 5 | 71.600000 | 0 | 1.193333 |
| L2 | 2 | 5 | 5 | 138.600000 | 0 | 2.310000 |
| L3 | 4 | 5 | 5 | 277.000000 | 0 | 4.616667 |

The rate estimate is calculated from the accepted transaction delta divided by the configured run duration. It should be interpreted as a run-window estimate, not as a precise steady-state throughput measurement.

## Success criteria evaluation

The v0.6.0 stage satisfies the following criteria:

1. The repository contains a dedicated v0.6.0-controlled-load-experiments branch.
2. The Dell server and Kubernetes preflight state were captured.
3. RPC access was validated through the Kubernetes solana-rpc service.
4. The load generator control path was validated through GET /stats and POST /rate.
5. An idle baseline run was captured.
6. Three controlled-load levels were executed.
7. Each controlled-load level was repeated five times.
8. All repeated controlled-load runs returned to target_lambda=0.
9. All repeated controlled-load runs ended with final_inflight=0.
10. The repeated controlled-load runs reported zero error delta.
11. Per-run and per-level summaries were generated.
12. The results were committed and pushed to the project repository.

## Interpretation

The results show that the testbed can execute repeated controlled-load transaction experiments at increasing commanded load levels.

Across L1, L2, and L3, all runs completed cleanly:

    err_delta_total=0
    clean_runs=5 for each controlled-load level

This supports the conclusion that the controlled-load workflow is repeatable under the tested conditions.

The observed results should not be interpreted as a saturation limit or maximum performance claim. The tested levels were intentionally conservative and were selected to validate repeatability rather than to stress the system to failure.

## Limitations

The main limitations are:

1. The clean-server deployment validation stage v0.5.1 remains deferred.
2. The experiments were executed on the existing Dell-based Kubernetes environment.
3. The current rate estimate is based on accepted transaction deltas over configured run duration and includes run-window overhead.
4. This stage does not identify the system saturation point.
5. This stage does not yet perform knee-point analysis.

## Conclusion

The v0.6.0 stage successfully establishes a repeatable controlled-load experiment workflow for the Solana Containerised Testbed.

The project can now proceed to v0.7.0, where the experimental objective changes from repeatability validation to saturation and knee-point analysis.
