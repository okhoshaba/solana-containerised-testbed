# v0.7.0 Saturation and Knee-Point Analysis

## Purpose

This document summarises the v0.7.0 saturation and knee-point analysis for the Solana Containerised Testbed.

The goal of this stage was to extend the repeated controlled-load experiments from v0.6.0 and identify the first observable saturation, instability, or knee-point boundary under progressively increasing transaction load.

## Methodological position

The experiment follows the project sequence:

    infrastructure -> observability -> controlled load -> saturation/knee analysis

The purpose of v0.7.0 was not to maximise benchmark numbers, but to determine where the current testbed begins to show non-clean behaviour under controlled load.

A run is treated as clean when:

    err_delta=0
    final_inflight=0

The field final_last_err is retained as diagnostic evidence, but it is not used as a mandatory clean/fail criterion because it can retain a previous error message after a later run completes without new errors.

## Experimental levels

The v0.7.0 experiment used the following load levels:

| Level | target_lambda | Interpretation |
|---|---:|---|
| S2 | 8 | low load |
| S3 | 16 | moderate low load |
| S4 | 32 | moderate load |
| S5 | 64 | elevated load |
| S6 | 128 | high load |
| S7 | 256 | boundary / post-incident validation zone |

Each level from S2 to S6 was repeated five times. S7 contains ordinary saturation-probe runs, one failed incident run, and one post-incident validation run.

## Clean operating region

The clean operating region extends through S6:

    S2 lambda=8
    S3 lambda=16
    S4 lambda=32
    S5 lambda=64
    S6 lambda=128

Across these levels:

- all runs were valid;
- all runs were clean;
- err_delta_total remained zero;
- final_inflight_mean remained zero;
- sample_err_per_sec_max remained zero;
- sample_inflight_max remained very low.

This means that no sustained saturation or queue accumulation was observed up to and including target_lambda=128.

## S7 boundary behaviour

S7 used target_lambda=256.

The first S7 pilot and subsequent early S7 repetitions showed that the testbed could process this load cleanly under some conditions. However, one S7 saturation-probe run exposed a validator-node storage-pressure incident.

The failed S7 run reported:

    sent_delta=3769
    ok_delta=3682
    err_delta=83
    final_inflight=4

The final error was:

    GetLatestBlockhash: rpc call getLatestBlockhash() on http://solana-rpc:8899: connect: connection refused

Kubernetes diagnostics showed that the validator pod was evicted because the validator node entered DiskPressure / ephemeral-storage pressure.

The failure path was:

    target_lambda=256
    validator node reaches storage pressure
    solana-validator-0 is evicted
    solana-rpc backend becomes unavailable
    loadgen2 reports connection refused
    err_delta becomes positive
    final_inflight remains non-zero

## Post-incident validation

After the storage-pressure incident, a storage preflight guard was added and the testbed was checked before running additional load.

The post-incident validation run at S7 target_lambda=256 completed without new errors:

    sent_delta=17465
    ok_delta=17465
    err_initial=87
    err_final=87
    err_delta=0
    final_inflight=0

The final_last_err field still retained the previous connection-refused error, but this was interpreted as a sticky diagnostic field rather than a new error because err_delta remained zero during the validation run.

The post-incident storage preflight also passed:

    DiskPressure=False
    validator pod Ready 1/1
    root_use_percent=38
    root_available_gb=28

This confirms that the testbed recovered after the incident.

## Knee-point interpretation

The v0.7.0 experiment did not identify a clean throughput knee in the strict sense.

Instead, it identified an infrastructure stability boundary at S7:

    clean stable region: S2-S6, up to target_lambda=128
    mixed boundary region: S7, target_lambda=256
    observed failure mode: validator-node storage pressure and pod eviction

Therefore, the current conclusion is:

    The first observed boundary is not a pure Solana/RPC throughput knee.
    It is a Kubernetes testbed storage-pressure boundary exposed during S7 load.

## Methodological consequence

Further high-load experiments, including S8 target_lambda=512, should not be executed until the storage design is improved.

Recommended mitigations include:

1. increasing disk capacity on the validator node;
2. moving local-path ledger storage to a larger filesystem;
3. moving containerd data-root to a larger filesystem;
4. keeping the storage preflight guard before high-load experiments;
5. adding explicit ephemeral-storage requests and limits;
6. distinguishing clean throughput saturation from infrastructure instability in future summaries.

## Final v0.7.0 conclusion

The v0.7.0 stage successfully extended the controlled-load methodology from v0.6.0 and identified the first meaningful boundary.

The testbed remained clean and stable through S6 target_lambda=128.

At S7 target_lambda=256, the experiment exposed a mixed boundary: several clean runs, one storage-pressure failure, and one successful post-incident validation run.

The appropriate conclusion is that v0.7.0 found a storage-pressure infrastructure boundary rather than a clean steady-state transaction throughput knee.
