# v0.7.0 Saturation and Knee-Point Analysis

## Purpose

This stage extends the repeated controlled-load workflow established in v0.6.0 toward saturation and knee-point analysis.

The purpose of v0.7.0 is to identify the region where increasing offered transaction load no longer produces proportional useful throughput and begins to cause disproportionate latency, errors, queueing, resource pressure, or instability.

This stage does not claim general Solana network performance. It analyses the behaviour of the Solana Containerised Testbed under the stated Dell, Kubernetes, container, RPC, and load-generator configuration.

## Starting point

The starting point is the completed v0.6.0 stage.

The v0.6.0 stage established:

- one idle baseline run;
- five clean repeated runs at lambda=1;
- five clean repeated runs at lambda=2;
- five clean repeated runs at lambda=4;
- zero error delta across the repeated controlled-load levels;
- consistent return to target_lambda=0 after each run;
- consistent final_inflight=0 after each run;
- per-run and per-level summaries.

Therefore, v0.7.0 can use lambda=1, 2, and 4 as validated low-load reference levels.

## Main research question

At what offered-load region does the testbed begin to show saturation or knee-point behaviour?

More specifically:

1. Does accepted throughput continue to grow proportionally as target_lambda increases?
2. Does err_delta remain zero at higher load levels?
3. Does final_inflight remain zero after controlled-load runs?
4. Does last_err remain empty?
5. Does the load generator report increasing inflight pressure during the run?
6. Do Kubernetes pods remain Running and Ready?
7. Do RPC checks remain healthy before and after the run?

## Methodological distinction from v0.6.0

The v0.6.0 stage tested repeatability under conservative load levels.

The v0.7.0 stage tests increasing offered load and looks for the first signs of non-linear behaviour.

Therefore:

- v0.6.0 emphasis: repeatability;
- v0.7.0 emphasis: saturation region and knee-point detection.

## Initial load schedule

The initial exploratory schedule is:

| Level | Target lambda | Meaning |
|---|---:|---|
| S0 | 0 | idle reference |
| S1 | 4 | validated elevated load from v0.6.0 |
| S2 | 8 | first higher-load step |
| S3 | 16 | moderate saturation probe |
| S4 | 32 | high saturation probe |
| S5 | 64 | conditional high-load probe |
| S6 | 128 | only if previous levels remain clean |

The schedule is intentionally conditional. The experiment should stop or pause if any of the following occur:

- err_delta becomes greater than zero;
- final_target_lambda does not return to zero;
- final_inflight remains greater than zero after cooldown;
- last_err is non-empty;
- the validator pod restarts;
- the loadgen2 pod restarts;
- RPC health check fails;
- Kubernetes reports CrashLoopBackOff, ImagePullBackOff, Pending, or Evicted states.

## Pilot strategy

The first v0.7.0 run should not jump directly to extreme load.

The recommended first pilot is:

- load_level=S2;
- target_lambda=8;
- duration_seconds=60;
- sample_interval_seconds=5;
- one pilot run only.

If S2 is clean, repeat S2 five times.

Only after S2 is clean should the experiment move to S3 with target_lambda=16.

## Candidate knee indicators

The knee point should not be inferred from one metric only.

Candidate indicators include:

- accepted transaction count stops growing proportionally with target_lambda;
- err_delta becomes positive;
- err_per_sec becomes positive during sampling;
- inflight increases and does not return quickly to zero;
- final_inflight remains positive;
- final_last_err is non-empty;
- RPC checks fail or slow down;
- Kubernetes pod state becomes unstable;
- resource pressure appears in host or pod telemetry.

## Conservative knee-point definition

For this project, the knee point is defined as the lowest tested offered-load level at which useful accepted throughput no longer increases proportionally while at least one independent stress signal appears.

The stress signal may be an error counter, inflight accumulation, RPC failure, pod instability, or clear resource pressure.

## Outputs

The expected v0.7.0 outputs are:

- raw run directories under results/v0.7.0/raw;
- processed per-run summary;
- processed per-level summary;
- saturation/knee-point summary;
- figures showing accepted count or rate versus target_lambda;
- figures showing error delta versus target_lambda;
- figures showing inflight behaviour versus target_lambda;
- final documentation report.

## Safety rule

Every controlled-load run must return loadgen2 to target_lambda=0 before the script exits.

This is mandatory even if the run fails.

## Limitation

The v0.7.0 results describe the saturation behaviour of this testbed configuration only. They do not represent the maximum performance of Solana as a public blockchain network.
