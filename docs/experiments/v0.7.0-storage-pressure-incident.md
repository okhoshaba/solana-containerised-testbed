# v0.7.0 S7 Storage Pressure Incident

## Summary

During the v0.7.0 saturation/knee-point analysis, the first instability boundary appeared at S7 with target_lambda=256.

This was not a clean steady-state throughput knee. The observed failure was caused by Kubernetes node storage pressure on the validator node.

## Affected run

Run:

    v0.7.0_coll_20260620T132552Z_S7-saturation-probe_lambda-256_ef156e7

Configured load:

    load_level=S7
    target_lambda=256
    duration_seconds=60

Observed run summary:

    sent_delta=3769
    ok_delta=3682
    err_delta=83
    final_target_lambda=0
    final_inflight=4

Final loadgen error:

    GetLatestBlockhash: rpc call getLatestBlockhash() on http://solana-rpc:8899: connect: connection refused

## Kubernetes evidence

The validator pod was evicted under node storage pressure.

The observed symptoms included:

    Evicted pod/solana-validator-0
    The node was low on resource: ephemeral-storage
    The node had condition: DiskPressure

The kubelet also reported image filesystem pressure:

    Disk usage on image filesystem is over the high threshold
    usage=85
    highThreshold=85
    lowThreshold=80

The kubelet attempted image garbage collection but failed to free sufficient space:

    Insufficient free disk space on the node's image filesystem
    85% of 47.9 GiB used

## Interpretation

The S7 failure should be interpreted as a testbed storage-pressure boundary rather than a pure Solana/RPC saturation knee.

The immediate failure path was:

    target_lambda=256
    validator node reaches DiskPressure
    solana-validator-0 is evicted
    solana-rpc backend becomes unavailable
    loadgen2 reports connection refused
    err_delta becomes positive
    final_inflight remains non-zero

## Methodological consequence

The current v0.7.0 evidence supports the following interpretation:

- S2 lambda=8 completed cleanly across five runs.
- S3 lambda=16 completed cleanly across five runs.
- S4 lambda=32 completed cleanly across five runs.
- S5 lambda=64 completed cleanly across five runs.
- S6 lambda=128 completed cleanly across five runs.
- S7 lambda=256 produced one clean pilot but later exposed a storage-pressure instability.

Therefore, the current knee-point candidate is not a clean throughput knee. It is an infrastructure stability boundary caused by node image-filesystem / ephemeral-storage pressure.

## Required mitigation before further high-load testing

Further S7 or S8 experiments should be paused until the validator node has sufficient storage headroom.

Recommended mitigation options:

1. Increase disk capacity of k8s-worker-01.
2. Move containerd data-root to a larger filesystem.
3. Move local-path/PVC ledger storage to a larger filesystem.
4. Clean unused container images and stale runtime data only after capturing diagnostics.
5. Add explicit ephemeral-storage requests/limits to make storage pressure visible earlier.
6. Add preflight checks that fail fast if image filesystem usage is close to kubelet eviction thresholds.

## Status

The incident diagnostics were captured under the failed run directory and committed to the v0.7.0 branch.

No further high-load experiments should be executed until storage pressure is mitigated.

## Post-incident storage footprint

A compact post-incident storage snapshot showed that the node later recovered to a non-pressure state.

At the time of the compact check, the root filesystem was approximately:

    48G total
    19G used
    28G available
    41% used

The largest persistent storage consumers observed were:

    /opt/local-path-provisioner      approximately 8.6G
    /var/lib/containerd              approximately 2.5G
    /var/lib/kubelet                 less than 1M

The validator ledger PVC was located under local-path provisioner storage:

    /opt/local-path-provisioner/pvc-..._validator-ledger-pvc

The largest subdirectory inside that PVC was:

    rocksdb                          approximately 4.4G

This suggests that the incident was related to temporary or threshold-sensitive node storage pressure involving the validator node image filesystem, container runtime storage, and local-path ledger storage, rather than large persistent pod logs.
