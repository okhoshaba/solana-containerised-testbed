# v0.7.1 High-Load Readiness Go/No-Go Decision

## Context

v0.7.0 found a clean stable region through S6 target_lambda=128.

At S7 target_lambda=256, the experiment exposed a mixed boundary. One run failed because the validator node entered DiskPressure, the validator pod was evicted, RPC became unavailable, and loadgen2 reported connection-refused errors.

## v0.7.1 storage inventory result

The v0.7.1 storage inventory showed that the validator node is currently healthy:

    DiskPressure=False
    root filesystem use approximately 36%
    root filesystem available space approximately 30G

The main storage consumers are:

    /opt/local-path-provisioner
    /var/lib/containerd

The validator ledger PVC is stored under /opt/local-path-provisioner. Container runtime data is stored under /var/lib/containerd. Both paths share the same root filesystem pressure domain.

## Decision

S8 target_lambda=512 is currently NO-GO.

## Reason

The v0.7.0 incident showed that the current storage layout can expose the validator pod to DiskPressure-driven eviction during high-load experiments.

Although the node is currently healthy, the storage design has not yet been changed. Therefore, S8 should not be attempted until storage capacity or storage isolation is improved.

## Required condition for GO

S8 may be reconsidered only after one of the following is completed:

1. a dedicated filesystem or disk is provided for local-path provisioner / validator ledger storage; or
2. the validator node root disk is expanded and the remaining shared-pressure risk is explicitly accepted.

In both cases, storage preflight must pass before any high-load run.

## Current conclusion

v0.7.1 confirms that the next research step should be storage redesign or explicit storage-capacity mitigation, not immediate S8 load generation.
