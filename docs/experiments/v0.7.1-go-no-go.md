# v0.7.1 Storage Redesign and High-Load Readiness: Go/No-Go Decision

## Purpose

This document records the minimal go/no-go decision for high-load continuation after the v0.7.0 saturation/knee-point analysis and the observed DiskPressure incident on the validator node.

The purpose of v0.7.1 is not to redesign the whole infrastructure immediately. Its purpose is to protect the research methodology before any higher-load experiment is attempted.

## Background

The v0.6.0 repeated controlled-load experiments and the v0.7.0 saturation/knee-point analysis established the following empirical interpretation:

* S2-S6, up to `target_lambda=128`, form the clean stable operating region.
* S7, `target_lambda=256`, is a mixed boundary region.
* The observed S7 failure was not interpreted as a clean steady-state transaction throughput knee.
* The failure was associated with infrastructure/storage pressure.
* One S7 run experienced DiskPressure on the validator node, eviction of `solana-validator-0`, temporary RPC unavailability, and `connection refused` errors.
* A later post-incident validation run completed without new errors.

The resulting interpretation is that v0.7.0 identified a storage-pressure infrastructure boundary rather than a pure transaction throughput knee.

## Storage Inventory Findings

The v0.7.1 storage inventory established that:

* the validator node is currently healthy;
* `DiskPressure=False`;
* the root filesystem is approximately 36% used at the time of the inventory snapshot;
* the validator ledger PVC is located under `/opt/local-path-provisioner`;
* containerd storage is located under `/var/lib/containerd`;
* both paths remain within the same root-filesystem pressure domain.

Therefore, the current deployment still has a shared storage-pressure risk between validator ledger growth, container runtime storage, images, logs, and other root filesystem activity.

## Current Decision

The current decision is:

```text
S8 target_lambda=512: NO-GO
```

This decision applies to the current storage layout.

The reason is that the previous S7 boundary already exposed infrastructure/storage-pressure behaviour. Running S8 without storage isolation or capacity mitigation would risk producing another infrastructure failure rather than scientifically useful transaction-load data.

## Conditions for S8 to Become GO

S8 may become GO only after at least one of the following mitigation paths is completed and documented.

### Option 1: Dedicated storage isolation

Provide a dedicated filesystem or disk for local-path provisioner / validator ledger storage.

This is the preferred option because it separates validator ledger growth from the root filesystem and from container runtime storage.

Minimum requirements:

* dedicated mount point for validator ledger PVCs;
* clear documentation of the mount path;
* confirmation that local-path provisioner uses the dedicated storage path;
* preflight check confirming sufficient free space before each high-load run.

### Option 2: Root disk capacity expansion

Expand the root disk capacity while explicitly accepting the residual shared-pressure risk.

This option is acceptable only as a temporary mitigation.

Minimum requirements:

* document the new root filesystem size;
* document free-space percentage before the high-load run;
* explicitly state that containerd and validator ledger storage still share the same pressure domain;
* perform storage preflight before each high-load run.

### Option 3: Mandatory high-load storage preflight

Before any high-load experiment at or above S7/S8, run a storage preflight check.

The preflight must verify at least:

* Kubernetes node condition `DiskPressure=False`;
* root filesystem usage;
* local-path provisioner usage;
* containerd usage;
* validator ledger PVC location;
* available free space;
* recent pod evictions;
* recent kubelet storage-pressure warnings.

## Research Consequence

The next research stage should not be S8.

The next stage should be:

```text
v0.8.0 dynamic load system identification
```

This stage should use the validated safe region as the primary operating range and should avoid the known storage-pressure boundary unless explicitly used as a controlled reference condition.

Recommended safe range for v0.8.0:

```text
target_lambda = 16, 32, 64, 128
```

Recommended boundary/reference condition:

```text
target_lambda = 256
```

Not recommended under the current storage layout:

```text
target_lambda = 512
```

## Final v0.7.1 Conclusion

The v0.7.1 conclusion is that the current infrastructure is suitable for continued dynamic system identification experiments inside the known safe operating region, but it is not yet suitable for uncontrolled escalation to S8.

The project should now merge v0.7.0 and v0.7.1 into `main`, then create a new branch for dynamic load system identification.
