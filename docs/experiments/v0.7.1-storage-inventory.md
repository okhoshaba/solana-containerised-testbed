# v0.7.1 Storage Inventory

## Purpose

This document summarises the first storage inventory captured during the v0.7.1 storage redesign and high-load readiness stage.

The purpose of this inventory is to explain the storage layout after the v0.7.0 S7 storage-pressure incident and to identify which storage paths should be considered before any further high-load experiments.

## Context

The v0.7.0 saturation and knee-point analysis found a clean stable region through S6 target_lambda=128.

At S7 target_lambda=256, the experiment exposed a mixed boundary region. One S7 run failed because the validator node entered DiskPressure / ephemeral-storage pressure. The validator pod was evicted, RPC became unavailable, and loadgen2 reported connection-refused errors.

The v0.7.1 stage therefore starts with storage inventory rather than new load generation.

## Inventory artifact

The inventory was captured using:

    scripts/load/10-v071-storage-inventory.sh

The snapshot is stored under:

    results/v0.7.1/storage-inventory/

## Current validator-node state

The validator node is:

    k8s-worker-01

At the time of inventory, the node was healthy:

    DiskPressure=False
    Ready=True

The root filesystem state was approximately:

    size: 48G
    used: 17G
    available: 30G
    use: 36%

Inode pressure was not observed.

## Main storage consumers

The main storage consumers on the validator node were:

    /opt/local-path-provisioner      approximately 6.4G
    /var/lib/containerd              approximately 2.5G
    /var/lib/kubelet                 less than 1M

The top-level filesystem layout showed:

    /opt                            approximately 6.6G
    /var                            approximately 3.0G
    /usr                            approximately 2.6G

It is bound to:

    pvc-1d0869fe-87d0-4f48-9602-28930cf7ee75

The declared capacity is:

The local-path storage location is:

    /opt/local-path-provisioner/pvc-1d0869fe-87d0-4f48-9602-28930cf7ee75_solana-observability_validator-ledger-pvc

The observed size was approximately:

This confirms that the validator ledger is stored through local-path provisioner under /opt.

## Container runtime storage

Container runtime storage is under:

    /var/lib/containerd

The observed size was approximately:

    2.5G

The largest containerd components were:

    /var/lib/containerd/io.containerd.snapshotter.v1.overlayfs    approximately 1.8G
    /var/lib/containerd/io.containerd.content.v1.content          approximately 654M

Containerd is not currently the dominant storage consumer, but it shares the same root filesystem pressure domain as /opt/local-path-provisioner.

## Interpretation

The current validator node is not under DiskPressure. However, the storage-pressure incident in v0.7.0 remains methodologically valid because it occurred during the S7 experiment sequence and caused validator pod eviction.

The current inventory shows that the two important storage paths are:

    /opt/local-path-provisioner
    /var/lib/containerd

Both are on the root filesystem. Therefore, ledger growth and container image/snapshot growth share the same eviction-pressure domain.

## Storage-redesign implication

The preferred redesign direction is to separate validator ledger storage from root filesystem pressure.

The strongest candidate is:

    attach or allocate a dedicated filesystem for local-path provisioner / validator ledger storage

A secondary option is:

    move containerd data-root to a separate filesystem

However, the inventory suggests that local-path ledger storage is the more important immediate target because it is the largest persistent project-specific storage consumer.

## Go/no-go implication for S8

S8 target_lambda=512 should not be attempted until the storage decision is explicit.

Minimum readiness conditions should include:

    DiskPressure=False
    no disk-pressure taint
    validator pod Ready 1/1
    root filesystem usage safely below threshold
    sufficient free space for ledger growth
    storage preflight passes
    local-path provisioner storage headroom is documented

## Conclusion

The v0.7.1 inventory confirms that the validator node is currently healthy, but the storage design still places the validator ledger PVC and container runtime data within the same root filesystem pressure domain.

Before further high-load testing, especially S8 target_lambda=512, the project should either redesign storage isolation or explicitly accept the current storage-pressure risk.
