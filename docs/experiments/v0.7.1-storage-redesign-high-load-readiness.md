# v0.7.1 Storage Redesign and High-Load Readiness

## Purpose

This stage addresses the storage-pressure boundary discovered during v0.7.0.

The v0.7.0 saturation and knee-point analysis showed clean behaviour through S6 target_lambda=128, but S7 target_lambda=256 exposed a validator-node storage-pressure incident. The validator pod was evicted under DiskPressure, causing RPC unavailability and loadgen connection-refused errors.

The purpose of v0.7.1 is to improve high-load readiness before attempting S8 target_lambda=512 or higher.

## Starting point

The starting point is the v0.7.0 result:

    clean stable region: S2-S6, up to target_lambda=128
    mixed boundary region: S7, target_lambda=256
    observed boundary: validator-node storage pressure
    failure mode: solana-validator-0 eviction and RPC unavailability

## Scope

v0.7.1 focuses on storage and high-load readiness only.

In scope:

- inspect validator-node storage layout;
- separate ledger/PVC storage from fragile root filesystem pressure;
- inspect containerd image and snapshot storage;
- retain the storage preflight guard;
- add storage-capacity documentation;
- define readiness criteria before further saturation testing.

Out of scope:

- S8 or higher load execution;
- MPC;
- reinforcement learning;
- Agave migration;
- unrelated Kubernetes redesign.

## Success criteria

v0.7.1 is successful when:

- the validator-node storage layout is documented;
- the source of v0.7.0 storage pressure is understood;
- the recommended storage redesign is implemented or explicitly deferred;
- the storage preflight guard is retained and documented;
- the testbed has a defined high-load readiness checklist;
- it is methodologically clear whether S8 can be attempted safely.
