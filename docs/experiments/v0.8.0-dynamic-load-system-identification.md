# v0.8.0 Dynamic Load System Identification

## Purpose

The purpose of v0.8.0 is to collect dynamic input-output datasets for the Solana/Kubernetes containerised testbed.

This stage is required before transfer-function identification, PID baseline design, MPC-oriented model design, frequency-response analysis, and anomaly detection based on controlled load excitation.

The goal is not to increase the maximum load. The goal is to excite the system in a controlled way inside the known safe operating region and measure the time-dependent response.

## Methodological Position

Previous stages established the empirical operating envelope.

v0.6.0 repeated controlled-load experiments and v0.7.0 saturation/knee-point analysis showed that:

- S2-S6, up to target_lambda=128, form the clean stable region.
- S7, target_lambda=256, is a mixed boundary region.
- The S7 incident exposed an infrastructure/storage-pressure boundary rather than a clean steady-state transaction throughput knee.
- S8, target_lambda=512, remains NO-GO under the current storage layout.

v0.8.0 therefore moves from static load characterisation to dynamic system identification.

The stage should answer:

- How does the system respond over time when target_lambda changes?
- What are the delay, gain, rise time, settling time, and recovery behaviour?
- How do latency, inflight transactions, errors, and RPC health evolve during transients?
- Can a repeatable healthy dynamic fingerprint be measured?
- Can deviations from that fingerprint indicate anomalies or early saturation?

## Control-System Formulation

The testbed is treated as a dynamic object of control:

    target_lambda(t) -> Solana/Kubernetes testbed -> y(t)

The controlled input is:

    u(t) = target_lambda(t)

The measured response is a vector:

    y(t) = [
      achieved_tps(t),
      ok_per_sec(t),
      err_per_sec(t),
      inflight(t),
      p50_latency(t),
      p95_latency(t),
      p99_latency(t),
      RPC health,
      RPC error counters,
      validator metrics,
      container metrics,
      pod metrics,
      node CPU/memory/disk/network metrics,
      Kubernetes events
    ]

For early transfer-function and PID-oriented work, the following SISO channels should be analysed first:

- target_lambda(t) -> achieved_tps(t)
- target_lambda(t) -> ok_per_sec(t)
- target_lambda(t) -> p95_latency(t)
- target_lambda(t) -> p99_latency(t)
- target_lambda(t) -> err_per_sec(t)
- target_lambda(t) -> inflight(t)

For later MPC-oriented work, these can be extended into MIMO, ARX, ARMAX, NARX, or state-space models.

## Confirmed Operating Limits

The safe baseline region for v0.8.0 is:

| Region | target_lambda | Status |
|---|---:|---|
| Low safe | 16 | GO |
| Medium safe | 32 | GO |
| Upper medium safe | 64 | GO |
| Upper clean stable | 128 | GO |
| Boundary/reference | 256 | CAUTIOUS REFERENCE ONLY |
| High-load S8 | 512 | NO-GO |

The default rule for dynamic excitation is:

    lambda_min = lambda_0 - A >= 0
    lambda_max = lambda_0 + A <= allowed_limit

The allowed limit depends on the experiment class:

| Experiment class | allowed_limit |
|---|---:|
| Safe baseline | 128 |
| Cautious extended probe | 160 or 192, only after safe-baseline success |
| Boundary reference | 256, only after explicit preflight and abort criteria |
| S8 high-load | NO-GO |

## Storage and Infrastructure Constraint

v0.7.1 established that the validator ledger PVC and containerd storage remain in the same root-filesystem pressure domain.

Therefore, v0.8.0 must not treat high-load failure as a clean transaction-processing limit unless infrastructure health is explicitly verified.

Before any v0.8.0 run, especially near or above target_lambda=128, the following must be checked:

- Kubernetes node condition DiskPressure=False.
- Root filesystem usage.
- local-path provisioner usage.
- containerd usage.
- validator ledger PVC location.
- recent pod evictions.
- recent kubelet storage-pressure warnings.
- validator pod status.
- RPC health.
- container restart counters.

S8 target_lambda=512 must not be executed in this stage.

## Experiment Sequence

The recommended methodological order is:

1. Baseline steady-state confirmation.
2. Step-response experiments.
3. Step-down and recovery experiments.
4. Ramp-up and ramp-down experiments.
5. Pulse and burst-recovery experiments.
6. Sinusoidal safe excitation.
7. Fourier-based frequency-response analysis.
8. Wavelet-based anomaly localisation.
9. PRBS-like and multi-level excitation.
10. Transfer-function, ARX, or state-space identification.
11. PID baseline design.
12. MPC-oriented prediction model design.

PID and MPC implementation are out of scope for v0.8.0. v0.8.0 produces the data needed for them.

## Experiment Class M5: Sinusoidal Load Excitation

Sinusoidal load excitation is used to estimate frequency response and to build a repeatable dynamic fingerprint of healthy testbed behaviour.

The load profile is:

    target_lambda(t) = lambda_0 + A * sin(2*pi*f*t)

where:

- lambda_0 is the mean load.
- A is the amplitude.
- f is the excitation frequency.
- T = 1/f is the period.
- t is experiment time.

### Scientific Purpose

Sinusoidal experiments are intended to:

- Estimate gain between input load and output metrics.
- Estimate phase shift and delay.
- Detect nonlinear behaviour through harmonics.
- Estimate noise floor.
- Create a healthy baseline dynamic fingerprint.
- Detect future anomalies as deviations from the baseline fingerprint.

### Initial Sine Profiles

| Profile | lambda_0 | A | Range | Status |
|---|---:|---:|---|---|
| SINE-A | 64 | 16 | 48-80 | SAFE BASELINE |
| SINE-B | 96 | 32 | 64-128 | SAFE BASELINE |
| SINE-C | 128 | 32 | 96-160 | CAUTIOUS EXTENDED |

SINE-C must not be run before successful SINE-A and SINE-B validation.

### Periods

| Period T | Description |
|---:|---|
| 300 s | slow oscillation |
| 180 s | medium-slow oscillation |
| 120 s | medium oscillation |
| 60 s | faster oscillation |

Each sine experiment should contain at least 5 full cycles.

10 cycles are preferred if the total duration remains reasonable.

Initial recommended sine runs:

| Run class | Profile | Period | Cycles | Status |
|---|---|---:|---:|---|
| M5-A-T120 | SINE-A | 120 s | 5 | GO |
| M5-A-T60 | SINE-A | 60 s | 5 | GO |
| M5-B-T120 | SINE-B | 120 s | 5 | GO |
| M5-B-T180 | SINE-B | 180 s | 5 | GO |
| M5-C-T120 | SINE-C | 120 s | 5 | AFTER SAFE VALIDATION |
| M5-C-T180 | SINE-C | 180 s | 5 | AFTER SAFE VALIDATION |

### Commanded vs Applied Input

The load generator should store both:

- target_lambda_commanded_continuous(t)
- target_lambda_applied_integer(t)

This is necessary because the mathematical sine profile may be continuous while the applied rate may be integer or discretised.

The applied input should be prevalidated before the run.

Clipping should not be used as a normal safety mechanism because clipping creates artificial harmonics and contaminates frequency-response analysis.

## Experiment Class M6: Chirp-Like Frequency Sweep

Chirp-like excitation should only be introduced after successful single-frequency sine validation.

Purpose:

- Estimate response across a range of frequencies.
- Identify frequency-dependent gain and delay.
- Detect bandwidth limits.
- Detect frequency ranges associated with latency amplification or instability.

Initial chirp limits:

- Keep lambda_max <= 128 for safe baseline chirp.
- Avoid chirp near boundary until safe chirp response is repeatable.
- Do not use chirp profiles that cross into S8 territory.

Status:

    OPTIONAL AFTER SINE VALIDATION

## Experiment Class M7: Optional Multisine Excitation

Multisine excitation may be used after single-sine and chirp-like experiments.

Purpose:

- Excite multiple frequencies simultaneously.
- Improve efficiency of frequency-response estimation.
- Test whether response remains approximately linear under combined periodic excitation.

Status:

    OPTIONAL, NOT INITIAL

Conditions:

- Single-frequency sine profiles must be repeatable.
- Harmonic distortion must be understood.
- Safe amplitude bounds must be validated.
- No boundary-reference multisine without explicit preflight.

## Experiment Class M8: PRBS-Like Safe Excitation

PRBS-like excitation is useful for ARX, ARMAX, NARX, and state-space identification.

Initial safe levels:

    target_lambda in {32, 64, 96, 128}

Alternative upper-safe set:

    target_lambda in {64, 96, 128}

The following should not be used initially:

    target_lambda in {64, 128, 256}

because it mixes identification with boundary-risk behaviour.

Purpose:

- Generate persistently exciting input.
- Support discrete-time model identification.
- Estimate multi-lag relationships.
- Support future MPC-oriented prediction models.

Status:

    AFTER STEP, RAMP, PULSE, AND SAFE SINE EXPERIMENTS

## Frequency-Response Analysis

For sinusoidal and chirp-like runs, perform frequency-domain analysis of input and output signals.

Required analyses:

- FFT of input signal.
- FFT of output signals.
- Gain estimation at excitation frequency.
- Phase shift estimation.
- Delay estimation.
- Harmonic detection at 2f, 3f, and 4f.
- Noise floor estimation.
- Comparison against healthy baseline response.

For a single excitation frequency f:

    gain_yu(f) = abs(Y(f)) / abs(U(f))

    phase_yu(f) = angle(Y(f)) - angle(U(f))

    delay_estimate = -phase_yu(f) / (2*pi*f)

The delay estimate must be interpreted carefully because of phase wrapping.

Practical requirements:

- Use an integer number of cycles.
- Preserve raw data.
- Preserve resampled data separately.
- Remove or mark warm-up intervals.
- Consider discarding the first cycle for steady periodic analysis.
- Verify timestamp alignment.
- Verify fixed sampling interval.
- Avoid clipped input profiles.

## Harmonic Distortion

If the input is a near-pure sine at frequency f, output components at 2f, 3f, and 4f may indicate nonlinear system behaviour.

Possible causes:

- Queueing nonlinearities.
- Latency amplification.
- Saturation onset.
- RPC instability.
- Transaction failure clustering.
- Resource contention.
- Storage or network jitter.
- Container/runtime interference.

Baseline harmonic indicators:

- H2/H1 ratio.
- H3/H1 ratio.
- H4/H1 ratio.
- Noise floor.
- Latency spike count.
- Error burst count.

These indicators can later be used for anomaly detection.

## Wavelet-Based Anomaly Localisation

Wavelet analysis is a supplementary time-frequency method.

It should be used to localise short-lived or non-stationary anomalies, including:

- latency spikes
- short overload bursts
- temporary RPC instability
- failed transaction clusters
- successful TPS drops
- disk I/O stalls
- network jitter episodes
- transition into saturation
- delayed recovery after overload

FFT answers which frequencies are present over the experiment.

Wavelet analysis helps identify when a local anomaly appears.

Expected outputs:

- time intervals with elevated time-frequency energy
- localised latency anomaly intervals
- localised error burst intervals
- localised RPC instability intervals
- comparison against healthy baseline wavelet signature

## Safety Gates

Before each dynamic run:

- Verify DiskPressure=False.
- Verify validator pod is running.
- Verify RPC endpoint is healthy.
- Verify no recent validator pod eviction.
- Verify no recent relevant kubelet storage warnings.
- Verify root filesystem usage.
- Verify local-path provisioner usage.
- Verify containerd usage.
- Verify free-space threshold.
- Verify node CPU/memory pressure status.
- Record preflight output into run metadata.

## Abort Conditions

A run must be aborted or marked invalid if any of the following occurs:

- DiskPressure=True.
- validator pod eviction.
- validator container restart during run.
- persistent RPC unavailable.
- sustained connection refused.
- rapidly increasing err_per_sec.
- unbounded inflight growth.
- severe latency runaway.
- disk free-space below threshold.
- Kubernetes reports relevant pressure condition.
- telemetry collection failure.
- load profile generator failure.

Abort thresholds should be declared before the run.

## Success Criteria

v0.8.0 is successful if the collected data allow the project to:

- describe the dynamic response to controlled load changes
- estimate delay, gain, settling, and recovery behaviour
- build an initial transfer function or discrete ARX/state-space model
- define safe controller operating limits
- obtain repeatable baseline dynamic response
- detect deviations from baseline in time domain
- detect deviations from baseline in frequency domain
- detect deviations from baseline in time-frequency domain
- separate normal operating region, near-boundary region, and unsafe/saturation behaviour
