# v0.8.0 Dynamic Load System Identification Run Matrix

## Purpose

This document defines the initial execution matrix for v0.8.0 dynamic load system identification.

The purpose is to ensure that experiments are executed in a controlled scientific order:

1. confirm the current healthy baseline;
2. collect simple dynamic response data;
3. collect recovery data;
4. collect periodic excitation data;
5. only then consider richer identification inputs such as PRBS-like, chirp-like, or multisine profiles.

This run matrix does not authorise S8 target_lambda=512.

## Global Safety Rule

The confirmed clean stable operating region is:

| Region | target_lambda | Status |
|---|---:|---|
| Low safe | 16 | GO |
| Medium safe | 32 | GO |
| Upper medium safe | 64 | GO |
| Upper clean stable | 128 | GO |
| Boundary/reference | 256 | CAUTIOUS REFERENCE ONLY |
| S8 high-load | 512 | NO-GO |

Dynamic profiles must satisfy:

    lambda_0 - A >= 0
    lambda_0 + A <= allowed_limit

For safe baseline experiments, allowed_limit is 128.

## Preflight Required Before Every Run

Before every v0.8.0 run, record:

- current git branch;
- current git commit SHA;
- Kubernetes node status;
- validator pod status;
- RPC health;
- DiskPressure condition;
- root filesystem usage;
- local-path provisioner usage;
- containerd usage;
- recent pod evictions;
- recent container restarts;
- recent kubelet storage-pressure warnings.

If DiskPressure=True, the run is NO-GO.

If the validator pod is not healthy, the run is NO-GO.

If RPC health is unstable before the run, the run is NO-GO.

## Phase 1: Step-Response Experiments

These runs estimate gain, delay, rise time, settling time, and transient behaviour.

| Order | Run ID | Profile | Status | Purpose |
|---:|---|---|---|---|
| 4 | M1-S32-64 | 32 -> 64 | GO | low-to-medium step |
| 5 | M1-S64-128 | 64 -> 128 | GO | medium-to-upper-safe step |
| 6 | M1-S32-128 | 32 -> 128 | GO after smaller steps | larger safe-region step |

Recommended segment duration:

- pre-step baseline: 5 min;
- post-step response: 10-15 min.

Required estimates:

- process gain;
- response delay;
- rise time;
- settling time;
- overshoot or undershoot;
- latency transient;
- inflight transient.

## Phase 6: Frequency-Response Analysis

This phase analyses sine runs before introducing more complex excitation.

Required outputs:

- FFT of input signal;
- FFT of output signals;
- gain at excitation frequency;
- phase shift;
- delay estimate;
- harmonic ratios H2/H1, H3/H1, H4/H1;
- noise floor estimate;
- comparison against baseline response.

Primary output channels:

- target_lambda(t) -> achieved_tps(t);
- target_lambda(t) -> ok_per_sec(t);
- target_lambda(t) -> p95_latency(t);
- target_lambda(t) -> p99_latency(t);
- target_lambda(t) -> err_per_sec(t);
- target_lambda(t) -> inflight(t).

## Phase 7: Wavelet-Based Anomaly Localisation

Wavelet analysis should be applied after safe step, pulse, ramp, and sine datasets exist.

Target anomaly classes:

- latency spikes;
- short overload bursts;
- temporary RPC instability;
- failed transaction clusters;
- successful TPS drops;
- disk I/O stalls;
- network jitter episodes;
- transition into saturation;
- delayed recovery after overload.

Required outputs:

- anomaly time intervals;
- affected metrics;
- time-frequency energy signature;
- relation to input load profile;
- run-level anomaly verdict.

## Phase 8: PRBS-Like Safe Excitation

PRBS-like excitation should not be the first dynamic experiment class.

Initial allowed levels:

    target_lambda in {32, 64, 96, 128}

Status:

    AFTER successful step, ramp, pulse, and sine validation

Purpose:

- ARX identification;
- state-space model identification;
- MPC-oriented prediction model design.

Do not initially use:

    target_lambda in {64, 128, 256}

because this mixes identification with boundary-risk behaviour.

## Phase 9: Optional Chirp and Multisine

Chirp-like and multisine runs are optional.

They should be introduced only after repeatable single-sine results.

Initial chirp rule:

    lambda_max <= 128

Boundary chirp is not part of the initial v0.8.0 run batch.

Multisine is not part of the initial v0.8.0 run batch.

## Explicit NO-GO Items

The following are not allowed in the initial v0.8.0 run matrix:

- S8 target_lambda=512;
- uncontrolled run above target_lambda=256;
- sine profile with lambda_max > 128 before safe sine validation;
- PRBS including 256 before safe PRBS validation;
- chirp crossing into boundary region before single-sine validation;
- storage migration inside this research branch;
- closed-loop PID control;
- MPC controller deployment.

## Success Criteria for the Initial Batch

The initial v0.8.0 batch is successful if it produces:

- complete load profile records;
- complete raw telemetry;
- aligned time series;
- run metadata;
- baseline steady-state summaries;
- step-response estimates;
- recovery estimates;
- sine frequency-response estimates;
- no DiskPressure incidents;
- no validator pod evictions;
- no uncontrolled RPC failure;
- no S8 execution.
