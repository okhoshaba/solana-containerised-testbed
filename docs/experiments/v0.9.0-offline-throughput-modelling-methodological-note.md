# v0.9.0 methodological note: offline throughput modelling

## Purpose

This note explains the methodological conclusion of the `v0.9.0 offline throughput modelling` stage.

The purpose of the stage is not to introduce a final controller yet. The purpose is to determine what kind of throughput-channel model is justified by the published v0.8.0 dynamic-identification dataset.

The analysed channel is:

    u_cmd(t) -> u_ach(t)

where:

- `u_cmd` is the commanded transaction load;
- `u_ach` is the achieved throughput.

    data/raw/v0.8.0/

The confirmed safe modelling range is:

    lambda = 32..128

## Main methodological conclusion

The current evidence supports a simple two-part throughput model.

### 1. Settled throughput model

For settled samples, the throughput channel is well approximated by a unity-gain model:

    u_ach[k] ~= u_cmd[k]

The settled-sample metrics are:

| metric | value |
|---|---:|
| RMSE | 0.094648 |
| MAE | 0.085025 |
| max absolute error | 0.293260 |
| mean error | 0.000204 |

This indicates that, in the confirmed safe range, the steady-state throughput response does not require a more complex static model.

### 2. Boundary-transition correction

For the first recorded sample after a commanded-load transition, the observed transition coefficient is:

    alpha = 0.856154

The corresponding first-order correction is:

    y[k] = y[k-1] + alpha * (u[k] - y[k-1])

or numerically:

    y[k] = y[k-1] + 0.856154 * (u[k] - y[k-1])

Equivalently:

    y[k] = 0.143846 * y[k-1] + 0.856154 * u[k]

This suggests that by the first recorded post-transition sample, approximately 85.6% of the throughput transition has already occurred.

## Consequence for PID and MPC preparation

For the initial PID/MPC research stage, the throughput plant can be treated as a near-unity-gain system in settled operation.

A minimal modelling assumption is therefore:

    plant gain K ~= 1

with a simple transition correction:

    alpha ~= 0.856

This is sufficient for initial controller prototyping, provided that the controller is evaluated against the same safe range:

    lambda = 32..128

The main control implication is that the controller should not overfit a complex steady-state model. The dominant observed modelling issue is not static gain error, but boundary-transition behaviour.

## What should not be concluded

The current evidence does not justify the following claims:

- that sub-5-second dynamics have been identified;
- that a latency-aware MPC model is ready;
- that the same model is valid outside `lambda = 32..128`;
- that `lambda = 512` can be safely included;
- that storage-pressure effects have been resolved;
- that a final PID or MPC design has already been validated.

## Latency limitation

Latency-constrained modelling remains out of scope.

The reason is methodological rather than theoretical: usable `lat_p99` values are missing in the current v0.8.0 `collect_csv.csv` outputs.

Therefore, the current v0.9.0 model is a throughput-channel model only.

A future latency-aware control stage requires fixing or extending telemetry collection before latency constraints can be included in MPC.

## Sampling limitation

Although the nominal collection sample interval is approximately 5 seconds, the transition-first-sample analysis observes a boundary interval of approximately 35 seconds across command changes.

Therefore, the transition coefficient should be interpreted carefully:

    alpha = 0.856154

does not identify sub-5-second dynamics. It describes the fraction of transition completed by the first recorded post-transition sample in the current dataset.

A future dynamic-identification experiment should use denser sampling around command boundaries if finer transfer-function estimation is required.

## Recommended next research step

The next step should be a controller-preparation stage, not immediate deployment of a closed-loop controller.

Recommended next stage:

    v0.10.0 throughput controller preparation

The stage should include:

- formalising the discrete-time throughput plant model;
- defining the controlled variable and manipulated variable;
- selecting safe command bounds;
- defining saturation and anti-windup policy;
- preparing a PID baseline design;
- preparing an MPC formulation using the simple throughput model;
- keeping latency constraints disabled until latency telemetry is fixed.

## Summary

The v0.9.0 offline modelling stage supports the following practical model:

    settled throughput:
        u_ach[k] ~= u_cmd[k]

    transition correction:
        y[k] = y[k-1] + 0.856154 * (u[k] - y[k-1])

This model is intentionally simple. That simplicity is a result of the data, not an assumption imposed before analysis.
