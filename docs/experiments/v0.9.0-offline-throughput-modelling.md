# v0.9.0 offline throughput modelling

## Purpose

This stage uses the published v0.8.0 dynamic load system-identification dataset to build an offline throughput-channel model for the Solana Containerised Testbed.

The modelling target is:

    u_cmd(t) -> u_ach(t)

where:

- `u_cmd(t)` is the commanded transaction load;
- `u_ach(t)` is the achieved throughput.

This stage does not run new Kubernetes experiments. It is an offline modelling stage based on the already published v0.8.0 dataset.

## Modelling scope

The current stage is throughput-only.

The latency channel is explicitly out of scope because `lat_p99` is missing in the v0.8.0 `collect_csv.csv` outputs.

## Candidate model classes

The first modelling pass should estimate and compare simple discrete-time throughput models.

### Model 0: static unity-gain baseline

    y[k] = u[k]

where:

- `u[k] = u_cmd[k]`
- `y[k] = u_ach[k]`

This model is the baseline implied by the v0.8.0 dynamic-identification summary.

### Model 1: static affine model

    y[k] = b0 + b1 u[k]

This model estimates steady-state gain and offset.

### Model 2: one-step delayed affine model

    y[k] = b0 + b1 u[k-1]

This model checks whether a one-sample command delay improves prediction.

### Model 3: first-order ARX model

    y[k] = a1 y[k-1] + b1 u[k-1] + b0

This model tests whether the throughput channel has measurable first-order dynamics at the current 5-second sampling resolution.

## Dataset split

Recommended first split:

- fit/calibration: M0, M1, M2, M3;
- validation: M4 sine-approximation.

Alternative robustness split:

- fit/calibration: M0, M1, M2;
- validation: M3 and M4.

## Evaluation metrics

Each candidate model should report:

- RMSE;
- MAE;
- max absolute error;
- mean signed error;
- R² where applicable;
- residual mean and residual standard deviation;
- per-run metrics;
- aggregate metrics.

## Expected scientific outcome

The v0.8.0 summary suggests that the throughput channel is close to unity gain in the confirmed safe range:

    lambda = 32..128

Therefore, the expected first result is that the static unity-gain baseline will already be strong.

The modelling script must verify this quantitatively rather than assuming it.

## Success criteria

This stage is successful if:

- all required v0.8.0 input CSV files are found;
- model fitting completes without errors;
- model metrics are generated per run and in aggregate;
- the best model is selected by validation RMSE/MAE;
- the final report clearly states whether a unity-gain model is sufficient at 5-second sampling resolution.

## Known limitations

- The modelling is throughput-only.
- Latency-constrained control remains out of scope.
- The current sampling interval is approximately 5 seconds.
- Sub-5-second transient dynamics cannot be identified from the current dataset.
