# Solana Containerised Testbed v0.3.0: Kubernetes Observability Core

## Release status

## DOI

Release notes for `v0.3.0`.

This release documents the migration of the existing Yellowstone/Geyser-enabled Observability Core from the Compose-based deployment model to a Kubernetes-based deployment model.

The release is focused on Kubernetes infrastructure, reproducibility, and validation of the observable Solana localnet environment.

## Summary

`v0.3.0` introduces a Kubernetes deployment layer for the Solana Containerised Testbed Observability Core.

The Compose-based architecture has been mapped to Kubernetes as follows:

```text
validator container      -> validator StatefulSet
validator ledger volume  -> PersistentVolumeClaim
wallet-init container    -> wallet-init Job
monitor container        -> monitor Deployment
RPC endpoint             -> solana-rpc Service on port 8899
Geyser gRPC endpoint     -> solana-geyser-grpc Service on port 10000
metrics endpoint         -> solana-metrics Service on port 9464


The Kubernetes deployment was validated on a CentOS 9 host using KVM and Minikube.

The validated infrastructure path was:

CentOS 9 host
  -> KVM
  -> Minikube VM
  -> single-node Kubernetes cluster
  -> Solana Observability Core

The host is a legacy x86_64 no-AVX2 server, which confirms the continued relevance of the no-AVX2 Solana compatibility images used by this repository.

## Main changes
Kubernetes scaffold

This release adds the initial Kubernetes structure:

k8s/
  base/
  overlays/
    minikube/
    kvm-multinode/

The base directory contains the portable Observability Core manifests.

The minikube overlay is used for the first validated Kubernetes deployment on the current CentOS/KVM host.

The kvm-multinode overlay is reserved for the future multi-node Kubernetes cluster based on dedicated KVM virtual machines.

## Validator StatefulSet

The Solana validator is deployed as a Kubernetes StatefulSet:

statefulset.apps/solana-validator

The validator uses the Yellowstone/Geyser-enabled no-AVX2 image:

docker.io/khoshaba/solana-localnet-validator:v1.18.25-noavx2-ivybridge-yellowstone

The validator exposes:

RPC:          8899
Geyser gRPC: 10000

Solana binaries inside the image are located under:

/opt/solana/bin

The Kubernetes probes therefore use the absolute Solana CLI path:

/opt/solana/bin/solana
Validator ledger PVC

The validator ledger is backed by a Kubernetes PersistentVolumeClaim:

validator-ledger-pvc

The validated Minikube deployment used:

StorageClass: standard
Capacity:     20Gi
Access mode:  ReadWriteOnce
RPC Service

The validator RPC endpoint is exposed internally through:

service/solana-rpc

Stable internal endpoint:

http://solana-rpc:8899
Yellowstone/Geyser gRPC Service

The Yellowstone/Geyser gRPC endpoint is exposed internally through:

service/solana-geyser-grpc

Stable internal endpoint:

solana-geyser-grpc:10000
Wallet initialisation Job

Wallet initialisation is implemented as a Kubernetes Job:

job.batch/wallet-init

The Job connects to the validator through:

http://solana-rpc:8899

and performs the initial wallet preparation required by the Observability Core.

## Scope boundaries

This release is intentionally limited to the Kubernetes Observability Core.

It does not introduce:

load generator
dashboard
Prometheus stack
Grafana
MPC controller
single-agent reinforcement learning
MARL
Agave research branch

Kubernetes is used here only as the hosting substrate for the observable Solana localnet system.

It is not yet used as an adaptive controller, learning policy, reinforcement-learning environment, or load-generation mechanism.

## Methodological position

This release belongs to the observation stage of the project roadmap.

The accepted methodological chain remains:

observation -> controlled load -> model -> MPC -> single-agent RL -> MARL

The Kubernetes Observability Core prepares the project for later controlled-load and controller-oriented stages, but those stages are not part of this release.

Known notes
Minikube and KVM

The first validation was performed on a single-node Minikube cluster running through the KVM2 driver.

This is not the final distributed Kubernetes architecture.

The future multi-node Kubernetes stage is expected to use dedicated headless Linux virtual machines under KVM/libvirt.

Storage

The Minikube virtual disk was located under:

/home/khoshaba/.minikube/machines/minikube/minikube.rawdisk

This keeps the main Minikube storage on the larger /home filesystem rather than the smaller root filesystem.

no-AVX2 compatibility

The host does not provide AVX2 CPU support.

This validates the continued need for the no-AVX2 Solana images used by this project.

Solana binary path

Inside the Kubernetes validator image, Solana binaries are installed under:

/opt/solana/bin

Validation scripts and Kubernetes probes should use absolute paths to avoid relying on shell PATH.

Reproducibility

The Kubernetes deployment can be applied with:

kubectl apply -k k8s/overlays/minikube

The system status can be checked with:

kubectl -n solana-observability get all
kubectl -n solana-observability get pvc
kubectl -n solana-observability get svc

RPC can be checked through port-forwarding:

kubectl -n solana-observability port-forward svc/solana-rpc 8899:8899

Metrics can be checked through port-forwarding:

kubectl -n solana-observability port-forward svc/solana-metrics 9464:9464

The wallet transfer validation can be executed with:

./scripts/k8s-wallet-transfer-validation.sh
