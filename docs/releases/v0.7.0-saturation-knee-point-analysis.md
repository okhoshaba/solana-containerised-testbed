# Solana Containerised Testbed v0.7.0

## Release title

Saturation and Knee-Point Analysis for the Solana Containerised Testbed v0.7.0

## Purpose

This release documents the v0.7.0 saturation and knee-point analysis stage of the Solana Containerised Testbed.

The goal was to extend the repeated controlled-load experiments from v0.6.0 and identify the first observable saturation, instability, or knee-point boundary under progressively increasing transaction load.

## Baseline

The starting point was the validated v0.6.0 repeated controlled-load experiment set, where controlled load levels were executed repeatedly and summarised using raw and processed experiment artifacts.

The v0.7.0 stage continued from that point with higher target_lambda levels and additional diagnostics.

## Main result

The testbed remained clean and stable up to:

    S6 target_lambda=128

The first non-clean boundary appeared at:

    S7 target_lambda=256

However, this boundary was not a clean steady-state throughput knee. It was an infrastructure stability boundary caused by validator-node storage pressure.

## Interpretation

The clean operating region is:

    S2-S6, up to target_lambda=128

The mixed boundary region is:

    S7, target_lambda=256

At S7, the experiment produced:

- three clean ordinary saturation-probe runs;
- one failed saturation-probe incident run;
- one clean post-incident validation run.

The failed S7 run exposed a Kubernetes storage-pressure failure mode:

    validator node DiskPressure
    solana-validator-0 eviction
    RPC backend unavailable
    loadgen2 connection refused
    err_delta=83
    final_inflight=4

## Storage-pressure incident

The S7 incident showed that the validator node reached DiskPressure / ephemeral-storage pressure. The validator pod was evicted, causing the RPC service backend to become temporarily unavailable.

This was documented in:

    docs/experiments/v0.7.0-storage-pressure-incident.md

The full saturation/knee analysis was documented in:

    docs/experiments/v0.7.0-saturation-knee-point-analysis.md

## Post-incident validation

After the incident, a storage preflight guard was added:

    scripts/load/09-v070-storage-preflight.sh

The post-incident validation run at S7 target_lambda=256 completed without new errors:

    err_delta=0
    final_inflight=0
    sample_err_per_sec_max=0

This showed that the testbed recovered after the storage-pressure incident, but it did not invalidate the earlier infrastructure-boundary observation.

## Methodological conclusion

v0.7.0 did not find a clean Solana/RPC throughput knee.

Instead, it found the first meaningful infrastructure boundary:

    clean stable region: S2-S6, up to target_lambda=128
    mixed boundary region: S7, target_lambda=256
    boundary type: validator-node storage pressure

Further S8 or higher-load experiments should be paused until storage design is improved.
