# Solana Containerised Testbed v0.8.0: Dynamic Load System Identification

## Release status

This release note describes the `v0.8.0 dynamic load system identification` stage of the Solana Containerised Testbed.

Persistent identifiers:

- Software DOI: 10.5281/zenodo.20828274
- Dataset DOI: 10.5281/zenodo.20834551

## Release purpose

The purpose of v0.8.0 is to extend the Solana Containerised Testbed with controlled dynamic-load experiments suitable for initial system identification.

The main identification channel is:

    u_cmd(t) -> u_ach(t)

where:

- `u_cmd(t)` is the commanded transaction load, represented by `target_lambda(t)` / `u_cmd`;
- `u_ach(t)` is the achieved throughput observed during the experiment.

This release focuses on throughput-channel identification. Latency-constrained control is explicitly deferred because `lat_p99` is missing in the current `collect_csv` outputs.

## Scientific scope

The v0.8.0 release covers the following experiment classes:

| class | purpose |
|---|---|
| M0 | steady-state baseline experiments |
| M1 | upward step-response experiments |
| M2 | step-down and recovery experiments |
| M3 | multi-step load-profile experiments |
| M4 | stepwise sine-approximation / initial frequency-response experiment |

The confirmed safe identification range is:

    lambda = 32..128

Higher-risk load levels are intentionally excluded from this release. In particular, `lambda=512` is not used.

## Main findings

The aggregate v0.8.0 dynamic-identification summary reports:

- all M0-M4 experiment classes completed successfully;
- the confirmed safe identification range is `lambda=32..128`;
- observed `err_per_sec` remained zero across the included experiment classes;
- throughput tracking is close to unity gain;
- step-like transitions settle within approximately one 5-second sample interval;
- the M4 sine-approximation run shows near-unity gain and very high correlation between commanded and achieved throughput.

Key M4 indicators:

- `gain_peak_to_peak = 1.002119`
- `gain_sinusoidal_fit = 0.998294`
- `zero_lag_correlation = 0.999539`
- `phase_lag_seconds = 1.223330`
- `RMSE = 0.997340 TPS`
- `MAE = 0.421263 TPS`

Overall conclusion:

    throughput_channel_ready_for_initial_modelling = true
    latency_channel_ready_for_modelling = false

## Known limitations

The main limitation is missing latency telemetry.

The current `collect_csv.csv` outputs do not contain usable `lat_p99` values. Therefore:

- v0.8.0 supports throughput-channel system identification;
- v0.8.0 does not yet support latency-constrained MPC;
- latency-aware modelling requires fixing or extending telemetry collection.

Additional limitations:

- settling-time estimates are limited by the current 5-second sampling resolution;
- the M4 sine experiment is a stepwise sine approximation, not a continuously varied sinusoidal command;
- the release is intentionally limited to `lambda=32..128`;
- `lambda=512` is excluded due to previously identified storage-pressure risk.
